네트워크 소켓 접속 흐름

핵심 요약: `GameServer`는 WinSock TCP 서버로 7777 포트를 열고, `DummyClient`는 로컬 서버에 접속하는 최소 네트워크 실행 경로를 제공합니다.

문서 위치: document/architecture/network-socket-flow.html · 작성 기준: 2026-06-18 로컬 변경사항

요약

서버 역할`GameServer`가 WinSock 초기화, listen 소켓 생성, 주소 바인딩, 접속 대기, accept를 담당합니다.
클라이언트 역할`DummyClient`가 루프백 주소 `127.0.0.1`의 7777 포트로 TCP 접속을 시도합니다.
통신 범위현재 범위는 연결 성립 확인까지입니다. 실제 패킷 송수신과 세션 생명주기는 아직 비어 있습니다.
실행 특성서버와 클라이언트 모두 블로킹 소켓 기반이며 무한 루프로 프로세스를 유지합니다.

목적과 책임

이 기능의 목적은 서버 코어 학습 프로젝트에 실제 TCP 접속 실험 지점을 만드는 것입니다. 이전의 메모리, 타입캐스팅, 스레드 샘플이 내부 코어 기능 검증에 가까웠다면, 이번 변경은 별도 실행 파일 두 개가 OS 네트워크 API를 통해 연결되는 외부 입출력 경계를 만듭니다.

GameServer/GameServer.cpp

서버 프로세스가 WinSock 2.2를 초기화하고 IPv4 TCP 소켓을 생성합니다. 이 소켓은 클라이언트와 직접 통신하는 소켓이 아니라 접속 요청을 받는 listen 소켓입니다.

serverAddr.sin_family = AF_INET;
serverAddr.sin_addr.s_addr = ::htonl(INADDR_ANY);
serverAddr.sin_port = ::htons(7777);

::bind(listenSocket, (SOCKADDR*)&serverAddr, sizeof(serverAddr));
::listen(listenSocket, 10);
GameServer/GameServer.cpp

`INADDR_ANY`는 서버 머신의 모든 IPv4 인터페이스에서 접속을 받을 수 있게 합니다. `htons(7777)`은 포트 값을 네트워크 바이트 오더로 변환해 서버와 클라이언트가 같은 포트 규약을 공유하게 합니다.

SOCKET clientSocket = ::accept(listenSocket, (SOCKADDR*)&clientAddr, &addrLen);

char ipAddress[16];
::inet_ntop(AF_INET, &clientAddr.sin_addr, ipAddress, sizeof(ipAddress));
cout << "Client Connected! IP = " << ipAddress << endl;
GameServer/GameServer.cpp

`accept`는 연결이 들어올 때까지 대기한 뒤, 실제 클라이언트와 통신할 별도 소켓을 반환합니다. IP 출력은 접속 성공을 관찰하기 위한 최소 로깅입니다.

::inet_pton(AF_INET, "127.0.0.1", &serverAddr.sin_addr);
serverAddr.sin_port = ::htons(7777);

if (::connect(clientSocket, (SOCKADDR*)&serverAddr, sizeof(serverAddr)) == SOCKET_ERROR)
    return 0;
DummyClient/DummyClient.cpp

클라이언트는 deprecated된 `inet_addr` 대신 `inet_pton`으로 문자열 IP를 바이너리 주소로 변환합니다. 목적지가 루프백 주소라서 서버와 클라이언트를 같은 개발 머신에서 바로 실행할 수 있습니다.

최근 변경

이번 diff에서 `GameServer`의 기존 RefCounting/Memory/TypeList/ThreadManager 실험 코드는 제거되고 TCP 서버 실행 코드로 대체되었습니다. `DummyClient`는 `HelloWorld()` 호출만 있던 상태에서 실제 WinSock 클라이언트로 확장되었습니다.

검증 포인트와 리스크

  • 서버를 먼저 실행한 뒤 클라이언트를 실행하면 서버 콘솔에 `Client Connected! IP = 127.0.0.1` 형태의 로그가 나와야 합니다.
  • 두 프로그램 모두 무한 루프가 있어 정상 종료 및 리소스 해제 코드는 현재 도달하기 어렵습니다.
  • 접속 성공 후 `clientSocket`을 저장하거나 닫는 로직이 없어, 반복 접속 테스트 시 리소스 관리 정책을 추가해야 합니다.
  • 이 단계는 블로킹 소켓 기반입니다. 이후 게임 서버 구조로 확장하려면 세션 객체, recv/send 루프, IOCP 또는 비동기 I/O 설계가 필요합니다.

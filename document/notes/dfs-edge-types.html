DFS 간선 4종류 — 특히 "교차 간선(cross edge)"

DeadLockProfiler 코드의 _finished[there] 분기가 뭘 거르는지 그림으로 정리합니다.

먼저 정정 한 가지 — 이전 요약 HTML에서 제가 _discoveredOrder[here] < _discoveredOrder[there]를 "교차 간선"이라고 적었는데, 이건 사실 전방 간선(forward edge)이에요. 아래에서 4가지를 한 번에 정리하면서 그 차이를 정확히 보여드릴게요.

1. 유향 그래프 DFS의 4가지 간선

트리 간선(Tree) 역방향 간선(Back) = 사이클 전방 간선(Forward) 교차 간선(Cross)
이름 discovered 비교 _finished[there] 의미 사이클?
트리 there == -1 (아직 미발견) — 처음 발견하는 노드로 내려가는 정상 진행 아님
역방향 here > there (there가 먼저 발견) false (DFS 진행 중) 콜스택에 살아있는 조상으로 거꾸로 가는 화살표 사이클!
전방 here < there (here가 먼저 발견) true 또는 false 둘 다 가능 here의 서브트리 안에 있는 후손으로 가는 지름길 아님
교차 here > there (there가 먼저 발견) true (DFS 이미 끝남) 나와 조상 관계가 아닌, 이미 끝난 다른 서브트리로 건너뛰는 화살표 아님

교차 간선은 "there가 나보다 먼저 발견됐지만, there의 DFS가 이미 완료된 상태"일 때만 생깁니다. 역방향(back)과 한 끗 차이고, 정확히 그 한 끗을 가르는 게 _finished[there]입니다.

2. 4가지 간선의 모양

트리 간선

새 노드를 처음 방문하며 따라가는 간선.

역방향 간선

사이클! A,B는 아직 DFS 진행 중.

전방 간선

조상이 후손을 직접 가리킴. 사이클 아님.

교차 간선

C는 이미 DFS 끝남. 조상도 후손도 아님.

3. 교차 간선 예시 ① — 같은 DFS 트리 안 형제 사이

다음 유향 그래프를 보세요.

노드 :   A, B, C, D
간선 :   A→B,  A→D,  B→C,  D→C

A에서 DFS를 시작하면 다음 그림과 같은 트리가 만들어집니다.

실행 순서

  1. Dfs(A) 시작 → discoveredOrder[A]=0
  2. A의 이웃 B 발견 → Dfs(B), discoveredOrder[B]=1
  3. B의 이웃 C 발견 → Dfs(C), discoveredOrder[C]=2
  4. C는 더 갈 데 없음 → finished[C]=true, return
  5. B도 더 갈 데 없음 → finished[B]=true, return
  6. 이제 A의 다른 이웃 D 발견 → Dfs(D), discoveredOrder[D]=3
  7. D→C 간선 검사 → 여기가 교차 간선!

코드의 if 분기로 따라가 보면

// here=D, there=C
if (_discoveredOrder[C] == -1)                // 2 != -1 → 통과
if (_discoveredOrder[D] < _discoveredOrder[C])  // 3 < 2? NO → continue 안 함
if (_finished[C] == false)                  // finished[C]=true → 진입 안 함
// → 어떤 분기에도 안 들어가고 자연스럽게 다음 이웃으로
// → 이게 바로 "교차 간선을 그냥 무시"하는 동작
왜 사이클이 아닌가? D가 C로 가는 화살표는 있지만, C는 이미 DFS가 끝나서 더 이상 누구도 기다리지 않습니다. C에서 D로 다시 돌아오는 경로가 없으니 사이클이 아니에요. 데드락 관점에서도, C 락은 잡았다가 이미 풀린 상태이므로 D가 C를 잡아도 서로 기다리지 않습니다.

4. 교차 간선 예시 ② — 서로 다른 DFS 트리 사이(포레스트)

DeadLockProfiler는 모든 노드에서 차례대로 DFS를 시작합니다. 

for (int32 lockId = 0; lockId < lockCount; lockId++)
    Dfs(lockId);

이러면 DFS 결과가 트리 하나가 아니라 여러 트리(=포레스트)가 됩니다. 이때 트리와 트리 사이를 잇는 간선이 모두 교차 간선이에요. 

노드 :   A, B, C
간선 :   A→B,  C→A

실행 순서

  1. Dfs(A) → A(0), B(1) 발견하고 둘 다 finished=true로 종료
  2. Dfs(B) → 이미 발견됨, 첫 줄에서 return
  3. Dfs(C) 시작 → C(2). C의 이웃 A 검사 → 교차 간선!
// here=C, there=A
if (_discoveredOrder[A] == -1)               // 0 != -1
if (_discoveredOrder[C] < _discoveredOrder[A]) // 2 < 0? NO
if (_finished[A] == false)                 // A의 DFS는 이미 끝남(true)
// → 그냥 다음 이웃으로 넘어감. 사이클로 신고 안 함.
핵심 — 형제 서브트리든 아예 다른 DFS 트리든, "다른 곳에서 시작해서 이미 끝난 노드"로 가는 간선은 전부 교차 간선이고, 사이클을 만들지 않습니다.

5. 코드의 if 분기 ↔ 4가지 간선 대응표

for (int32 there : nextSet) {
    if (_discoveredOrder[there] == -1) {        // ① 트리 간선
        _parent[there] = here;
        Dfs(there);
        continue;
    }
    if (_discoveredOrder[here] < _discoveredOrder[there])
        continue;                                // ② 전방 간선 무시

    if (_finished[there] == false) {              // ③ 역방향 = 사이클!
        printf("%s -> %s\n", ...);
        CRASH("DEADLOCK_DETECTED");
    }
    // ④ (암묵적 else) → 교차 간선: 아무것도 안 하고 다음 이웃
}
번호코드 분기간선 종류처리
①_discoveredOrder[there] == -1 트리재귀 진입
②here < there 전방continue (무시)
③_finished[there] == false 역방향CRASH (데드락)
④위 어디에도 안 걸림 (암묵적 else) 교차아무 동작 없음
그래서 답 — 교차 간선은 _finished[there] == true인 케이스예요. 코드에는 그 분기가 따로 적혀있지 않고, 위 3개 if를 다 빠져나오는 모든 경우가 곧 교차 간선입니다. DFS 입장에서는 "이미 끝난 다른 서브트리에 화살표를 쏘는 것"이라 무시해도 안전합니다.

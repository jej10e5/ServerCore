StompAllocator와 Virtual Memory 정리

페이지 단위 가상 메모리를 이용해서 메모리 오염을 빨리 발견하는 디버그 allocator

큰 그림 Virtual Memory 현재 코드 Stomp 아이디어 잡는 것과 못 잡는 것 요약

1. StompAllocator가 하려는 일

BaseAllocator는 malloc/free를 얇게 감싼 일반 할당기다. 반면 StompAllocator는 성능 좋은 allocator가 아니라, 메모리 버그를 더 빨리, 더 크게 터뜨리기 위한 디버그용 allocator다.

BaseAllocator

ServerCore/Allocator.cpp

필요한 크기만큼 비교적 촘촘하게 메모리를 잡는다. 효율적이지만, 작은 메모리 오염이 바로 티 나지 않을 수 있다.

StompAllocator

ServerCore/Allocator.h

아주 작은 객체도 OS 페이지 단위로 크게 잡는다. 메모리를 낭비하지만, 디버그 중 버그를 찾기 쉬워진다.

이름의 느낌은 “잘못 밟으면 크게 소리 나게 만드는 allocator”에 가깝다. 평소에는 조용히 지나갈 수 있는 메모리 실수를 디버그 단계에서 빨리 발견하려는 장치다.

2. 운영체제 Virtual Memory부터 잡기

프로그램이 보는 주소는 실제 RAM 주소가 아니다. 프로그램은 자기만의 넓은 가상 주소 공간을 보는 척하고, 운영체제와 CPU의 MMU가 이 가상 주소를 실제 물리 메모리로 매핑한다.

Virtual Address

프로그램 코드에서 보는 주소다. 포인터 값이 여기에 해당한다.

Physical Memory

실제 RAM이다. 프로그램은 보통 이 주소를 직접 다루지 않는다.

Page

가상 메모리를 관리하는 기본 단위다. Windows에서 흔히 4KB, 즉 0x1000 바이트다.

페이지 단위로 관리된다는 뜻

OS는 메모리를 바이트 하나씩 보호하지 않고, 보통 페이지 단위로 보호한다. 어떤 페이지는 읽기/쓰기 가능, 어떤 페이지는 접근 금지, 어떤 페이지는 아직 실제 RAM에 연결되지 않은 상태일 수 있다.

Page 00x1000 bytesREAD/WRITE
Page 10x1000 bytesREAD/WRITE
Page 20x1000 bytesNO ACCESS
Page 30x1000 bytesReserved

Reserve와 Commit

용어 의미 비유
MEM_RESERVE 가상 주소 공간 일부를 찜해둔다. 다른 할당이 이 범위를 쓰지 못하게 한다. 좌석을 예약만 해둔 상태
MEM_COMMIT 실제로 접근 가능한 메모리로 준비한다. 물리 메모리 또는 페이지 파일 backing이 붙는다. 예약한 좌석에 실제로 앉을 수 있게 준비된 상태
PAGE_READWRITE 해당 페이지를 읽고 쓸 수 있게 보호 속성을 지정한다. 출입 가능 표지판

3. 현재 StompAllocator 코드 읽기

현재 Debug 빌드에서는 CoreMacro.h에서 xxalloc이 StompAllocator::Alloc으로 연결된다. Release에서는 다시 BaseAllocator를 사용한다. 

// CoreMacro.h
#ifdef _DEBUG
#define xxalloc(size)   StompAllocator::Alloc(size)
#define xxrelease(ptr)  StompAllocator::Release(ptr)
#else
#define xxalloc(size)   BaseAllocator::Alloc(size)
#define xxrelease(ptr)  BaseAllocator::Release(ptr)
#endif

Alloc

void* StompAllocator::Alloc(int32 size)
{
    const int64 pageCount = (size + PAGE_SIZE - 1) / PAGE_SIZE;
    const int64 dataOffset = pageCount * PAGE_SIZE - size;

    void* baseAddress = ::VirtualAlloc(
        NULL,
        pageCount * PAGE_SIZE,
        MEM_RESERVE | MEM_COMMIT,
        PAGE_READWRITE);

    return static_cast<void*>(
        static_cast<int8*>(baseAddress) + dataOffset);
}

여기서 가장 중요한 줄은 dataOffset = pageCount * PAGE_SIZE - size다. 이 계산 때문에 객체 시작 주소가 페이지의 맨 앞이 아니라 페이지 끝에 딱 붙는 위치가 된다.

dataOffset을 그림으로 보기

VirtualAlloc이 돌려준 baseAddress는 페이지의 시작 주소다. 그런데 StompAllocator는 이 주소를 그대로 반환하지 않는다. 앞쪽에 일부러 여백을 두고, 객체가 들어갈 시작 위치까지 포인터를 밀어서 반환한다.

핵심 아이디어: 객체를 페이지 시작에 두는 것이 아니라, 객체의 끝을 페이지 끝에 맞춘다. 즉 반환 포인터는 baseAddress가 아니라 baseAddress + dataOffset이다. 이렇게 하면 객체 뒤쪽으로 넘쳐 쓰는 실수를 페이지 경계 근처로 몰아넣을 수 있고, guard page까지 붙인 구현에서는 1바이트 초과 접근도 바로 접근 위반으로 잡을 수 있다.
핵심을 한 줄 그림으로 다시 보기 일반 배치: [ Object ][ 남는 공간 .................... ] | 페이지 끝 StompAllocator 배치: [ 남는 공간 .................... ][ Object ] | 페이지 끝 ^ 반환 포인터 목표: Object 뒤쪽 overflow를 페이지 경계 쪽으로 밀어붙인다.

Release

void StompAllocator::Release(void* ptr)
{
    const int64 address = reinterpret_cast<int64>(ptr);
    const int64 baseAddress = address - (address % PAGE_SIZE);

    ::VirtualFree(reinterpret_cast<void*>(baseAddress), 0, MEM_RELEASE);
}

xdelete가 받은 포인터는 페이지 시작 주소가 아니라 객체 시작 주소다. 그래서 address % PAGE_SIZE를 빼서 페이지 시작 주소를 다시 계산한 뒤 VirtualFree에 넘긴다.

4. 왜 객체를 페이지 끝에 붙이는가?

메모리 버그 중 흔한 것은 “내가 받은 객체 범위를 넘어서 쓰는 것”이다. 예를 들어 100바이트만 할당받았는데 101번째 바이트에 쓰는 식이다. 일반 allocator에서는 바로 뒤에 다른 객체나 allocator metadata가 있을 수 있어서, 당장 크래시가 안 나고 나중에 엉뚱한 곳에서 터질 수 있다.

일반 allocator 느낌

[ Object A ][ Object B ][ Object C ] ^ A가 넘쳐 쓰면 B를 조금 망가뜨릴 수 있음

오염은 발생했지만 즉시 발견되지 않을 수 있다.

StompAllocator 느낌

[ 빈 공간 ][ Object A ] | 페이지 경계 ^ A의 끝을 경계에 붙임

객체 끝 근처의 오버런을 페이지 경계와 연결해서 더 빨리 드러내려는 구조다.

엄밀히 말하면 현재 코드만으로는 다음 페이지를 PAGE_NOACCESS로 만드는 guard page가 없다. 그래서 “객체 뒤쪽 오버플로우를 100% 즉시 잡는다”기보다는, 페이지 단위 배치와 해제를 이용해 메모리 오염, 잘못된 해제, 해제 후 사용 같은 문제를 더 거칠게 드러내려는 출발점으로 이해하면 좋다.

더 강한 StompAllocator는 보통 guard page를 둔다

실무에서 흔히 말하는 stomp/guard allocator는 객체 바로 뒤 페이지를 접근 금지로 만들어둔다. 그러면 객체 범위를 1바이트만 넘어 써도 다음 페이지 접근에서 access violation이 발생한다.

강한 guard page 방식의 개념 [ READ/WRITE 페이지: 객체를 끝에 배치 ][ NO ACCESS 페이지 ] Object 끝 | 1바이트 넘어 쓰면 즉시 크래시

5. 이 allocator가 잡는 것과 못 잡는 것

문제 유형 현재 구현에서의 기대 설명
잘못된 free 주소 잡힐 가능성이 높아짐 페이지 시작 주소 계산과 VirtualFree 요구 조건 때문에 잘못된 포인터가 더 쉽게 문제를 일으킨다.
Use After Free 잡힐 가능성이 높아짐 VirtualFree 후 해당 가상 메모리 범위는 더 이상 유효하지 않으므로 접근 시 크래시가 날 수 있다.
객체 뒤쪽 overflow 현재 구현만으로는 제한적 다음 페이지를 접근 금지로 만들지 않았기 때문에 항상 즉시 잡히지는 않는다.
객체 앞쪽 underflow 잘 못 잡음 객체 앞에는 같은 페이지의 여백이 있으므로 앞쪽으로 조금 침범해도 접근 가능한 메모리일 수 있다.
메모리 사용량 매우 비효율적 작은 객체 하나에도 최소 4KB 페이지가 필요하다. 디버그 용도로만 적합하다.

6. 꼭 가져가야 할 감각

1. 포인터는 가상 주소다

내 코드가 보는 주소는 OS가 관리하는 가상 주소 공간 안의 주소다.

2. OS는 페이지 단위로 보호한다

읽기/쓰기 가능, 접근 금지, 예약, 커밋 같은 속성은 페이지 단위로 작동한다.

3. StompAllocator는 디버그 도구다

메모리를 아끼려는 목적이 아니라, 메모리 버그를 빨리 드러내려는 목적이다.

4. 페이지 끝 배치가 핵심이다

객체를 페이지 끝에 붙이면 범위를 넘는 쓰기와 페이지 경계를 연결할 수 있다.

한 문장으로 말하면, StompAllocator는 OS의 가상 메모리 페이지라는 큰 단위를 이용해서 작은 메모리 실수를 크게 보이게 만드는 디버그용 할당기다.

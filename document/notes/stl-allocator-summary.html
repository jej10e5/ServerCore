STL Allocator 실습 정리

STL 컨테이너가 내부 메모리 할당을 내 allocator 경로로 보내는 구조 이해하기

목표 StlAllocator 필수 규격 타입 변환 Container alias Map 포인트 체크리스트

1. STL Allocator를 만드는 이유

vector, map, string 같은 STL 컨테이너는 내부에서 원소 저장 공간을 직접 할당한다. 기본 상태에서는 std::allocator를 사용한다. 이번 실습의 목표는 이 기본 allocator 대신 서버 코어의 xxalloc/xxrelease 경로를 태우는 것이다.

기본 STL 컨테이너 vector<int> -> std::allocator<int> -> operator new / delete 계열 이번 실습 컨테이너 Vector<int> -> StlAllocator<int> -> xxalloc / xxrelease -> Debug: StompAllocator -> Release: BaseAllocator
핵심은 STL 컨테이너의 메모리 할당 정책을 우리 쪽 allocator 정책으로 바꾸는 것이다. 컨테이너 사용법은 거의 그대로 두고, 내부 할당 경로만 바꾼다.

2. 현재 StlAllocator 코드

ServerCore/Allocator.h 
template<typename T>
class StlAllocator
{
public:
    using value_type = T;

    StlAllocator() {}

    template<typename Other>
    StlAllocator(const StlAllocator<Other>&) {}

    T* allocate(size_t count)
    {
        const int32 size = static_cast<int32>(count * sizeof(T));
        return static_cast<T*>(xxalloc(size));
    }

    void deallocate(T* ptr, size_t count)
    {
        xxrelease(ptr);
    }
};
구성 요소 역할
value_type 이 allocator가 할당하는 원소 타입을 STL에게 알려준다.
기본 생성자 컨테이너가 allocator 객체를 기본 생성할 수 있게 한다.
변환 생성자 StlAllocator<T>를 StlAllocator<Other>로 바꿔 쓸 수 있게 한다.
allocate(count) count개의 T가 들어갈 원시 메모리를 할당한다.
deallocate(ptr, count) 할당했던 원시 메모리를 해제한다. 현재 구현에서는 count를 쓰지 않는다.

3. 꼭 알아야 하는 STL allocator 규격

1. allocator는 타입별 템플릿이다

StlAllocator<int>와 StlAllocator<Knight>는 서로 다른 타입이다. STL 컨테이너는 원소 타입에 맞는 allocator 타입을 요구한다.

2. allocate는 생성자를 호출하지 않는다

allocate는 메모리 공간만 준다. 원소의 생성자 호출과 소멸자 호출은 컨테이너가 따로 처리한다.

3. count는 바이트가 아니라 원소 개수다

allocate(10)은 10바이트가 아니라 T 10개 공간이라는 뜻이다. 그래서 count * sizeof(T) 계산이 필요하다.

4. 컨테이너 내부 타입도 할당한다

STL은 원소만 할당하지 않는다. 노드, proxy, bucket 등 내부 관리 객체를 위해 다른 타입의 allocator도 필요로 할 수 있다.

4. 막히기 쉬운 핵심: 다른 타입 allocator로 변환

STL 컨테이너는 내부 구현상 StlAllocator<T>만 쓰지 않는다. 예를 들어 vector<int32, StlAllocator<int32>>라도 Visual Studio STL 내부에서 std::_Container_proxy 같은 관리 객체용 allocator가 필요할 수 있다.

vector<int32, StlAllocator<int32>> 원소 저장 공간: StlAllocator<int32> 내부 관리 객체: StlAllocator<std::_Container_proxy> 그래서 STL은 allocator 타입을 다른 T로 바꿔 쓸 수 있어야 한다.

이때 필요한 것이 변환 생성자다. 

template<typename Other>
StlAllocator(const StlAllocator<Other>&) {}
실습 중 막혔던 지점은 StlAllocator<Other&>처럼 Other에 참조를 붙이는 형태였다. STL이 넘기는 것은 보통 StlAllocator<Other>이지 StlAllocator<Other&>가 아니므로 변환 생성자를 찾지 못한다.

5. Container.h alias의 의미

ServerCore/Container.h

매번 vector<Type, StlAllocator<Type>>처럼 쓰면 길고 실수하기 쉽다. 그래서 Container.h에서 서버용 컨테이너 별칭을 만든다. 

template<typename Type>
using Vector = vector<Type, StlAllocator<Type>>;

template<typename Key, typename Type, typename Pred = less<Key>>
using Map = map<Key, Type, Pred, StlAllocator<pair<const Key, Type>>>;
사용 코드 Map<int32, Knight> m; m[100] = Knight(); 실제 타입 느낌 std::map< int32, Knight, std::less<int32>, StlAllocator<std::pair<const int32, Knight>> >
앞으로 서버 코드에서는 STL 원본 타입보다 Vector, Map, String 같은 alias를 쓰면 컨테이너 내부 메모리 할당이 서버 코어 allocator 정책을 탄다.

6. Map에서 pair<const Key, Type>가 중요한 이유

std::map<Key, Type>의 실제 원소 타입은 pair<const Key, Type>다. key는 map 안에서 정렬 기준이 되므로, 원소로 들어간 뒤 마음대로 바뀌면 트리 구조가 깨진다. 그래서 key 부분이 const다.

컨테이너 allocator value_type 이유
Vector<T> T 원소가 그대로 T다.
List<T> T 컨테이너가 내부 노드 allocator로 변환해서 쓴다.
Map<K, V> pair<const K, V> map의 원소 타입 자체가 key/value pair다.
String char 문자 배열 저장 공간을 allocator로 잡는다.
주의할 점: allocator의 value_type은 컨테이너가 기대하는 원소 타입과 맞아야 한다. 특히 set/unordered_set 계열은 const Key를 allocator 타입에 넣고 싶은 유혹이 있지만, 컨테이너가 요구하는 value_type과 정확히 맞는지 확인해야 한다.

7. allocate/deallocate와 생성자/소멸자의 분리

xnew/xdelete에서는 직접 placement new와 명시적 소멸자 호출을 했다. 하지만 STL allocator의 allocate/deallocate는 원시 메모리만 담당한다. 객체 생성/소멸은 STL 컨테이너가 allocator traits를 통해 처리한다.

xnew/xdelete xnew: xxalloc(sizeof(T)) placement new로 생성자 호출 xdelete: 소멸자 직접 호출 xxrelease STL allocator allocate: xxalloc(count * sizeof(T)) vector/map 내부: 필요한 시점에 생성자 호출 필요한 시점에 소멸자 호출 deallocate: xxrelease
이 차이를 구분하는 것이 중요하다. StlAllocator::allocate 안에서 placement new를 직접 호출하면 STL 컨테이너가 다시 생성자를 호출할 수 있어 구조가 꼬인다.

8. 실습 체크리스트

필수value_type

using value_type = T;가 있어야 STL이 allocator의 원소 타입을 알 수 있다.

필수변환 생성자

StlAllocator<Other>에서 변환 가능해야 내부 타입 allocator로 바꿔 쓸 수 있다.

필수count 의미

allocate(count)의 count는 바이트가 아니라 원소 개수다.

주의생성자 호출 금지

allocate는 메모리만 잡고, 객체 생성은 컨테이너에게 맡긴다.

주의Map value_type

Map<K,V>의 allocator 타입은 pair<const K,V> 기준이다.

디버그xxalloc 경로

Debug에서는 StompAllocator, Release에서는 BaseAllocator로 연결된다.

한 문장으로 정리하면, StlAllocator는 STL 컨테이너가 요구하는 최소 규격을 맞추면서, 실제 메모리 확보/해제만 서버 코어의 allocator 정책으로 보내는 어댑터다.

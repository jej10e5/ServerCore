운영체제 Virtual Memory 자세히 보기

프로세스가 보는 주소와 실제 RAM 사이에서 OS와 CPU가 무엇을 하는지 정리

왜 필요한가 주소 공간 페이지 주소 변환 Page Fault Reserve/Commit 보호 요약

1. Virtual Memory는 왜 필요한가?

프로그램에서 포인터를 보면 마치 메모리 주소를 직접 다루는 것처럼 느껴진다. 하지만 현대 OS에서 프로세스가 보는 주소는 실제 RAM 주소가 아니라 가상 주소다.

격리

프로세스 A가 프로세스 B의 메모리를 마음대로 건드리지 못하게 한다.

편의

각 프로세스는 자신만의 넓고 연속적인 주소 공간을 가진 것처럼 코드를 작성할 수 있다.

효율

필요한 부분만 실제 메모리에 올리고, 파일이나 디스크 공간과 연결해 사용할 수 있다.

핵심 감각: 포인터 값은 실제 RAM 칩의 위치가 아니라, 현재 프로세스의 가상 주소 공간 안에서의 위치다.

2. 프로세스마다 독립적인 주소 공간이 있다

같은 숫자 주소라도 프로세스마다 의미가 다를 수 있다. 예를 들어 A 프로세스의 0x000001234000과 B 프로세스의 0x000001234000은 같은 숫자지만 서로 다른 메모리를 가리킬 수 있다.

Process A virtual address space 0x0000 ---------------- [ code ][ heap ][ stack ] ---------------- 0xFFFF... Process B virtual address space 0x0000 ---------------- [ code ][ heap ][ stack ] ---------------- 0xFFFF... 같은 가상 주소 숫자라도 각 프로세스의 page table이 다르면 실제 연결 대상이 다르다.

이 독립성 덕분에 한 프로세스의 포인터 버그가 곧바로 다른 프로세스의 메모리를 망가뜨리지 않는다. OS는 프로세스마다 별도의 주소 변환 정보를 관리한다.

3. Virtual Memory의 기본 단위: Page

OS는 가상 메모리를 바이트 하나씩 관리하지 않는다. 보통 페이지라는 고정 크기 블록 단위로 관리한다. Windows/x64 환경에서 흔히 보는 기본 페이지 크기는 4KB, 즉 0x1000 바이트다.

Page 04KBREAD/WRITE
Page 14KBREAD/WRITE
Page 24KBNO ACCESS
Page 34KBRESERVED

페이지 단위라는 사실은 매우 중요하다. 접근 권한, 실제 메모리 연결, 디스크로 밀려나는 단위, page fault 처리 단위가 모두 페이지를 기준으로 움직인다.

4. 가상 주소는 어떻게 실제 메모리로 바뀌는가?

CPU가 메모리에 접근하려고 하면, MMU라는 하드웨어가 가상 주소를 물리 주소로 변환한다. 이때 OS가 만들어 둔 page table을 참고한다.

Virtual Address 프로그램이 사용하는 포인터 값
예: 0x7FF6_1234_5678
--->
Page Table 가상 페이지가 어느 물리 프레임에 연결되는지 기록한 표
Physical Address 실제 RAM의 위치
<---
MMU CPU 안에서 주소 변환과 권한 검사를 수행하는 하드웨어
가상 주소의 대략적인 분해 [ virtual page number ][ page offset ] | | | +-- 페이지 안에서 몇 번째 바이트인지 | +-- page table에서 어떤 물리 프레임인지 찾는 데 사용 결과: [ physical frame number ][ 같은 page offset ]

페이지 안의 offset은 변하지 않는다. 바뀌는 것은 “어느 페이지인가”에 해당하는 앞부분이다. 그래서 page table은 가상 페이지 번호를 물리 프레임 번호로 매핑하는 표라고 생각하면 된다.

5. Page Fault는 무조건 오류가 아니다

Page fault라는 말을 보면 곧바로 크래시를 떠올리기 쉽지만, 사실 page fault는 “CPU가 접근하려는 페이지를 현재 상태로는 바로 처리할 수 없다”는 신호에 가깝다. OS가 처리할 수 있으면 정상적으로 계속 진행되고, 처리할 수 없으면 접근 위반이 된다.

상황 OS의 처리 결과
페이지가 아직 RAM에 올라와 있지 않음 디스크나 파일에서 페이지를 가져온다. 프로그램은 잠깐 멈췄다가 계속 실행
예약만 되어 있고 commit되지 않음 접근을 허용할 수 없다. 접근 위반 가능
읽기 전용 페이지에 쓰기 시도 권한 검사 실패 접근 위반 또는 copy-on-write 처리
전혀 유효하지 않은 주소 접근 매핑 정보가 없음 크래시
그래서 page fault는 “사건”이고, access violation은 “OS가 이 사건을 정상 처리할 수 없을 때의 결과”라고 보면 좋다.

6. Reserve와 Commit

Windows의 VirtualAlloc을 보면 MEM_RESERVE, MEM_COMMIT이라는 단어가 자주 나온다. 이 둘은 비슷해 보이지만 역할이 다르다.

상태 의미 접근 가능 여부
Reserved 가상 주소 범위를 찜해둔다. 다른 할당이 이 범위를 쓰지 못한다. 아직 접근하면 안 됨
Committed OS가 이 페이지에 실제 backing을 제공하기로 약속한 상태다. 보호 속성이 허용하면 접근 가능
Free 아무것도 예약되지 않은 가상 주소 공간이다. 접근 불가
비유 Reserve = 이 땅은 내가 쓸 거니까 다른 사람은 쓰지 마. Commit = 이제 이 땅에 실제로 건물을 세우고 사용할 수 있게 해줘. Protect = 이 건물은 읽기만 가능 / 읽고 쓰기 가능 / 출입 금지.

7. Protection: 페이지별 접근 권한

가상 메모리의 큰 장점은 페이지마다 접근 권한을 줄 수 있다는 점이다. 이 기능 덕분에 스택 가드, 실행 방지, 읽기 전용 코드 영역, guard page 같은 기법이 가능해진다.

권한 의미 예시
PAGE_READWRITE 읽기와 쓰기가 가능하다. heap, 일반 데이터
PAGE_READONLY 읽기는 가능하지만 쓰기는 불가능하다. 상수 데이터, 일부 매핑 파일
PAGE_NOACCESS 읽기, 쓰기 모두 불가능하다. guard page, 접근 감지용 페이지
실행 권한 코드를 실행할 수 있는지 제어한다. DEP/NX 같은 보안 기능
allocator 디버깅에서 guard page를 쓰는 이유가 여기에 있다. 객체 뒤에 PAGE_NOACCESS 페이지를 두면, 객체 범위를 넘어 쓰는 순간 OS가 접근 위반으로 알려줄 수 있다.

8. Stack, Heap, Code도 결국 가상 메모리 위에 있다

우리가 평소 말하는 stack, heap, code 영역도 프로세스의 가상 주소 공간 안에 배치된 영역들이다. 차이는 누가 관리하고 어떤 용도로 쓰느냐에 있다.

영역 주 용도 특징
Code 실행할 명령어 보통 읽기/실행 가능, 쓰기는 제한
Stack 함수 호출, 지역 변수 스레드마다 존재, guard page로 stack overflow 감지 가능
Heap 동적 할당 new, malloc, allocator가 주로 사용
Mapped File 파일을 메모리처럼 접근 파일 내용과 가상 메모리 페이지를 연결

9. 꼭 기억할 핵심

포인터는 가상 주소다

실제 RAM 주소가 아니라 현재 프로세스의 주소 공간 안에서 해석되는 값이다.

page table이 번역한다

가상 페이지 번호를 물리 프레임 번호로 바꾸는 표가 있고, CPU의 MMU가 이를 사용한다.

OS는 페이지 단위로 관리한다

보호, commit, page fault, swap 같은 동작은 대부분 페이지 단위로 일어난다.

page fault는 처리 가능한 사건일 수 있다

모든 page fault가 크래시는 아니다. OS가 처리하지 못할 때 access violation이 된다.

한 문장으로 정리하면, Virtual Memory는 각 프로세스에게 독립적인 주소 공간을 제공하고, OS와 CPU가 페이지 단위로 실제 메모리와 연결하며 보호하는 시스템이다.

배경 Set / Release lvalue vs rvalue 복사 생성자 이동 생성자 소멸자와의 관계 나란히 비교 함정 모음

0. 왜 이게 필요한지부터

TSharedPtr는 참조 카운트(reference count)로 메모리를 관리하는 스마트 포인터입니다. 원시 포인터(T*)와 달리, 몇 명이 이 객체를 "소유"하고 있는지를 세고 마지막 소유자가 떠날 때 자동으로 delete를 호출합니다.

관리 규칙 한 줄 요약
  • "누군가 이 포인터를 받으면" → refCount ++
  • "누군가 이 포인터를 놓으면" → refCount --
  • refCount == 0 이 되는 순간 → delete

그런데 C++에서 객체를 "받는" 방법은 여러 가지입니다. 복사 생성자는 살아있는 객체로부터, 이동 생성자는 곧 사라질 임시 객체로부터 받습니다. 두 경우는 refCount를 처리하는 방식이 완전히 다르기 때문에 함수를 따로 만들어야 합니다.

코드 전체 구조 훑기

ServerCore/RefCounting.h
// ── 관리되는 객체의 베이스 클래스 ──
class RefCountable {
    atomic<int32> _refCount = 1;   // 생성 시 자신이 첫 소유자
    int32 AddRef()    { return ++_refCount; }
    int32 ReleaseRef() {
        int32 n = --_refCount;
        if (n == 0) delete this;   // 마지막 소유자 → 자살
        return n;
    }
};

// ── 스마트 포인터 ──
template<typename T>
class TSharedPtr {
    T* _ptr = nullptr;

    // 내부 헬퍼
    void Set(T* ptr);     // _ptr 저장 + AddRef
    void Release();      // ReleaseRef + _ptr = nullptr

    // Rule of 5
    TSharedPtr(const TSharedPtr& rhs);   // 복사 생성자
    TSharedPtr(TSharedPtr&& rhs);         // 이동 생성자
    ~TSharedPtr();
    TSharedPtr& operator=(const TSharedPtr&);
    TSharedPtr& operator=(TSharedPtr&&);
};

1. Set / Release — 모든 것의 기반

복사·이동 생성자를 이해하기 전에 이 두 헬퍼를 먼저 봐야 합니다. 이 둘이 모든 refCount 조작을 담당합니다.

Set()

void Set(T* ptr) {
    _ptr = ptr;
    if (ptr)
        ptr->AddRef(); // refCount ++
}
  • 포인터를 _ptr에 저장
  • nullptr가 아니면 AddRef() → "내가 소유자가 됨"을 등록

Release()

void Release() {
    if (_ptr != nullptr) {
        _ptr->ReleaseRef(); // refCount --
        _ptr = nullptr;    // 내 손 놓기
    }
}
  • nullptr 체크로 이중 해제 방어
  • ReleaseRef() 안에서 ref가 0이 되면 자동 delete
  • 반드시 _ptr = nullptr로 마무리
핵심 원칙
소유하면 Set, 포기하면 Release. 이 두 함수만 잘 이해하면 복사/이동/대입/소멸자 모두 같은 패턴이에요.

2. lvalue vs rvalue — 시그니처가 갈리는 이유

복사 생성자와 이동 생성자가 따로 있는 근본 이유는 인자의 종류가 다르기 때문입니다.

종류 정의 예시 참조 타입 누가 받는가
lvalue 이름이 있는 값.
이 줄이 끝나도 살아있음		 
TSharedPtr<Player> a(...);
 const T& 복사 생성자
rvalue 이름이 없는 값.
이 줄 끝나면 사라짐		 
TSharedPtr<Player>(new Player())
std::move(a)
 T&& 이동 생성자
왜 구분해야 하나?
  • lvalue는 복사 후에도 원본이 계속 쓰인다 → 내가 ref를 +1 해야 한다.
  • rvalue는 어차피 곧 소멸한다 → ref를 올렸다가 바로 내리는 낭비 없이, 포인터만 훔쳐오면 된다.

컴파일러가 자동으로 선택한다

TSharedPtr<Player> a(new Player()); // a는 lvalue

TSharedPtr<Player> b = a;               // → 복사 생성자  (a는 lvalue)
TSharedPtr<Player> c = std::move(a);    // → 이동 생성자  (move(a)는 rvalue)
TSharedPtr<Player> d = GetPlayer();     // → 이동 생성자  (반환값은 rvalue)

두 시그니처만 만들어 두면, 컴파일러가 인자 타입에 따라 자동으로 알맞은 오버로드를 고릅니다.

3. 복사 생성자 — 완전 해부

3-1. 시그니처를 토큰 단위로 뜯기

타입 키워드 lvalue 참조 & 매개변수 이름
토큰왜 이런 모양?
TSharedPtr 생성자 이름 = 클래스 이름. 반환 타입 없음.
이 함수는 새 TSharedPtr 객체를 만드는 역할.
const rhs를 읽기만 하고 절대 수정하지 않겠다는 약속.
복사할 때는 원본(rhs)에 손댈 이유가 없으니까 반드시 붙임.
덕분에 const TSharedPtr<Player>&도 인자로 받을 수 있다.
TSharedPtr& lvalue 참조. &가 하나인 것이 핵심.
이름 있는 값(lvalue)만 이 참조에 바인딩 된다.
&가 없으면 인자가 복사돼서 무한 재귀가 발생함.
TSharedPtr(TSharedPtr rhs) — ❌ 위험
이 형태는 함수 호출 시 rhs를 복사하려는데, 그 복사가 또 이 함수를 부름 → 무한 재귀
rhs "right-hand side"의 약자. 대입 기준으로 오른쪽 피연산자. TSharedPtr b = a에서 a가 rhs에 바인딩.

3-2. 본문 동작 — 한 줄이지만

TSharedPtr(const TSharedPtr& rhs) {
    Set(rhs._ptr);   // ← 이게 전부
}

Set(rhs._ptr) 안에서 실제로 일어나는 일:

  1. _ptr = rhs._ptr
    나(this)의 포인터를 rhs와 같은 주소로 설정한다. 이제 둘이 같은 Player 객체를 가리킨다.
  2. rhs._ptr→AddRef()
    그 Player 객체의 refCount를 +1 한다. "나도 이 객체의 소유자"라고 등록하는 것.

3-4. 왜 필요한가

없으면 어떻게 되나?

복사 생성자를 안 만들면 컴파일러가 얕은 복사(shallow copy)를 기본으로 제공합니다. 얕은 복사는 _ptr 값만 그대로 복사하고 AddRef()를 호출하지 않아요. 

// 컴파일러 기본 복사 (AddRef 없음)
b._ptr = a._ptr;  // ← refCount는 여전히 1

// 나중에 b가 먼저 소멸하면
b.~TSharedPtr() → Release() → refCount 1→0 → delete Player!
// a는 아직 살아있는데 Player는 이미 사라짐 → 💀 use-after-free

4. 이동 생성자 — 완전 해부

4-1. 시그니처를 토큰 단위로 뜯기

ServerCore/RefCounting.h : 42
TSharedPtr ( TSharedPtr && rhs )
타입 rvalue 참조 && 매개변수 이름
토큰왜 이런 모양?
TSharedPtr 생성자 이름 = 클래스 이름. 복사 생성자와 동일.
TSharedPtr&& rvalue 참조. &가 두 개인 것이 핵심.
이름 없는 임시값(rvalue)만 이 참조에 바인딩 된다.
lvalue를 여기에 넘기려면 std::move()로 강제 변환해야 함.
std::move(a)는 a를 실제로 이동시키는 게 아닙니다.
"이 값은 lvalue지만 rvalue처럼 취급해도 돼"라고 컴파일러에게 알려주는 캐스트일 뿐. 실제 이동은 이동 생성자/대입 연산자가 한다.
const 없음! 복사 생성자와 달리 const가 없습니다.
이동 생성자는 반드시 rhs를 수정해야 하기 때문입니다. (rhs._ptr = nullptr)
const가 붙으면 rhs._ptr을 바꿀 수 없어서 컴파일 에러.
rhs 복사 생성자와 동일한 이름. 여기서는 "곧 죽을 rvalue 객체"를 가리킴.

4-2. 본문 동작 — 두 줄의 의미

TSharedPtr(TSharedPtr&& rhs) {
    _ptr = rhs._ptr;      // ① 포인터를 훔쳐온다
    rhs._ptr = nullptr;   // ② 원본을 비워서 무력화한다
}
  1. _ptr = rhs._ptr — 포인터를 훔쳐온다
    rhs가 갖고 있던 포인터 주소를 내 것(this->_ptr)으로 가져온다. 이 시점에서 AddRef()가 없다. rhs의 소유권을 그대로 이전받는 것이므로 refCount를 건드릴 필요가 없다.
  2. rhs._ptr = nullptr — 원본을 무력화한다
    rhs는 곧 소멸한다. 소멸자는 Release()를 부르는데, nullptr이면 Release()가 아무것도 하지 않는다. 이걸 안 하면 rhs의 소멸자가 내가 지금 들고 있는 객체를 delete해 버린다.
    nullptr 처리 없이 rhs가 소멸하면:
    ~TSharedPtr() → Release() → refCount-- → 0 → delete!
    나(this)는 이미 그 주소를 들고 있는데 → use-after-free

4-4. 왜 필요한가

이동 생성자 없으면 생기는 문제

이동 생성자가 없으면 컴파일러는 rvalue 인자도 복사 생성자로 처리합니다.

// 이 코드를 실행했을 때
pool.Add(TSharedPtr<Player>(new Player()));
//       ^^^^^^^^^^^^^^^^^^^^^^^^^ rvalue (임시값)
이동 생성자 없을 때 (복사로 처리)
  1. 임시 TSharedPtr 생성 → ref = 1
  2. 복사 생성자 호출 → AddRef() → ref = 2
  3. 임시값 소멸 → Release() → ref = 1
atomic 연산이 2번 발생. 멀티스레드에서 atomic은 비싸다.
이동 생성자 있을 때
  1. 임시 TSharedPtr 생성 → ref = 1
  2. 이동 생성자 호출 → 포인터만 이전
  3. 임시값 소멸 → _ptr==nullptr → 아무것도 안 함
atomic 연산 0번. 성능 이득.

5. 소멸자와의 관계 — 전체 생명주기

~TSharedPtr() { Release(); }

소멸자는 단순히 Release()를 호출합니다. 이 단순함이 유지되는 이유는 이동 생성자에서 rhs._ptr = nullptr을 보장하기 때문입니다. rhs가 소멸될 때 _ptr이 nullptr이라 Release()가 아무것도 하지 않는 거예요.

전체 생명주기 예시

void Example() {
    TSharedPtr<Player> a(new Player());   // Set(ptr) → ref=1
    {
        TSharedPtr<Player> b = a;           // 복사 → ref=2
        TSharedPtr<Player> c = std::move(a); // 이동 → ref=2, a=null

        // b 소멸: Release() → ref=1
        // c 소멸: Release() → ref=0 → delete Player!
    }
    // a.Release(): _ptr==nullptr이므로 아무것도 안 함
}
관찰 포인트
  • 복사(b = a)는 ref를 2로 올렸으므로, b 하나가 사라져도 Player는 살아있다.
  • 이동(c = std::move(a))은 ref를 건드리지 않았으므로 여전히 ref = 2.
  • b가 소멸 → ref 1. c가 소멸 → ref 0 → delete Player.
  • a는 이동 후 nullptr이 되어 소멸해도 아무 일도 안 일어난다.

6. 복사 vs 이동 나란히 비교

복사 생성자 이동 생성자
시그니처 TSharedPtr(const TSharedPtr& rhs) TSharedPtr(TSharedPtr&& rhs)
const? 있음 — rhs 변경 안 함 없음 — rhs 변경해야 함
받는 인자 lvalue (이름 있는 값) rvalue (이름 없는 임시값)
AddRef 호출 있음 — ref ++ 없음
rhs 상태 변화 그대로 — 원본 유지 nullptr — 무력화
의미 소유권을 공유 소유권을 이전
atomic 비용 1회 (AddRef) 0회
호출 예 
TSharedPtr b = a;
func(a);
 
TSharedPtr b = std::move(a);
func(std::move(a));
TSharedPtr b = GetPlayer();

7. 자주 빠지는 함정들

① 이동 후 원본 사용

TSharedPtr<Player> a(new Player());
TSharedPtr<Player> b = std::move(a);
a->DoSomething();  // ❌ a._ptr은 nullptr! 역참조 크래시
std::move(a)로 이동한 뒤 a는 nullptr 상태입니다. 이동 후에는 원본을 사용하면 안 됩니다. 컴파일러는 이 오류를 잡아주지 않아요.

② nullptr 처리를 빠뜨리면

// 이렇게 짜면 안 됩니다
TSharedPtr(TSharedPtr&& rhs) {
    _ptr = rhs._ptr;
    // rhs._ptr = nullptr; ← 이걸 빠뜨리면
}
// rhs 소멸 시 Release() → refCount 0 → delete → b는 dangling pointer!

③ & 하나와 && 둘의 혼동

void Foo(TSharedPtr& sp)  { ... }  // lvalue 참조만 받음
void Bar(TSharedPtr&& sp) { ... }  // rvalue 참조만 받음

TSharedPtr a(new Player());
Foo(a);              // OK
Bar(std::move(a));   // OK
Foo(std::move(a));   // ❌ 컴파일 에러 — rvalue를 lvalue ref로 못 받음
Bar(a);              // ❌ 컴파일 에러 — lvalue를 rvalue ref로 못 받음

④ 변환 생성자의 타입 오타

// 현재 코드 (오타 의심)
template<typename U>
TSharedPtr(const TSharedPtr<U>& rhs) {
    Set(static_cast<T>(rhs._ptr));   // ← T 여야 하나 T* 여야 하나?
}

// 올바른 형태
Set(static_cast<T*>(rhs._ptr));     // ← 포인터 캐스팅이어야 함
static_cast<T>는 객체 타입으로 변환을 시도하는데, 포인터를 객체로 변환할 수 없어 대부분의 경우 컴파일 에러가 납니다. static_cast<T*>가 의도된 형태입니다.
세 줄 요약
  • 복사 생성자: const T& — 살아있는 값, 소유권 공유, AddRef
  • 이동 생성자: T&& + const 없음 — 임시값, 소유권 이전, rhs=nullptr
  • 둘의 차이를 만드는 핵심은 & 하나냐 && 둘이냐 — 이게 인자 종류(lvalue/rvalue)를 가른다

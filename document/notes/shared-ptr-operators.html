TSharedPtr 연산자 총정리

복사·이동·복사대입·이동대입·변환 생성자까지, 한 클래스 안에서 다 만나는 케이스 정리

왜 이렇게 많지? Rule of 5 L-value vs R-value 복사 생성자 이동 생성자 변환 생성자 복사 대입 이동 대입 기타 연산자 최종 정리표

0. 왜 이렇게 많은 함수가 필요할까?

TSharedPtr은 참조 카운트라는 자원을 직접 관리합니다. 누가 사본을 만들면 카운트를 +1 해줘야 하고, 죽으면 -1 해줘야 하며, 그러다 0이 되면 delete this를 해야 해요.

그런데 C++에서 객체를 만드는 방법은 한 가지가 아닙니다. "누가, 어떤 종류의 값을 통해 초기화·대입하느냐"에 따라 다른 함수가 호출돼요. 그래서 한 클래스에 "복사", "이동", "복사 대입", "이동 대입"이 따로 필요해집니다.

이걸 C++에서는 Rule of 5라고 부르고, TSharedPtr도 정확히 그 5개를 다 갖춰놨습니다.

1. Rule of 5 — 자원 관리 클래스의 5종 세트

#이름시그니처언제 호출?
1소멸자 ~TSharedPtr() 스코프 종료 / delete 시
2복사 생성자 TSharedPtr(const TSharedPtr&) 새 객체를 다른 객체로 초기화
3이동 생성자 TSharedPtr(TSharedPtr&&) 새 객체를 임시값(rvalue)으로 초기화
4복사 대입 연산자 operator=(const TSharedPtr&) 기존 객체에 다른 객체 값을 덮어쓰기
5이동 대입 연산자 operator=(TSharedPtr&&) 기존 객체에 임시값을 덮어쓰기
꼭 기억할 차이 — "생성자"는 지금 막 태어나는 객체가 호출하고, "대입 연산자"는 이미 살아있는 객체가 호출합니다. 그래서 대입 연산자는 항상 먼저 자기가 갖고 있던 자원을 정리(Release)해야 합니다.

2. 그 전에 — L-value 와 R-value

복사/이동 두 버전이 따로 있는 이유는 결국 인자가 "이름이 있는 값" 이냐 "곧 사라질 임시값" 이냐에 따라 다르게 처리하기 위해서예요.

구분표기예시호출되는 것
lvalue 이름 있는 값 const T& TSharedPtr b = a; 복사 생성자
rvalue 임시값 T&& TSharedPtr b = std::move(a);
return TSharedPtr(p);		 이동 생성자

컴파일러가 이 둘 중 어느 오버로드를 부를지 자동으로 골라줍니다. 우리는 두 시그니처를 모두 만들어 두기만 하면 돼요.

3. 복사 생성자 — TSharedPtr(const TSharedPtr& rhs)

ServerCore/RefCounting.h : 40
TSharedPtr(const TSharedPtr& rhs) { Set(rhs._ptr); }

의미

다른 TSharedPtr을 보고 새 객체를 만들 때 호출됩니다. 같은 raw 포인터를 가리키게 한 뒤 refCount를 +1 해주는 게 핵심.

호출 예시

TSharedPtr<Player> a(new Player());   // ref=1
TSharedPtr<Player> b = a;                // 복사 생성자 → ref=2
TSharedPtr<Player> c(a);                 // 동일

그림으로

핵심 — 복사 생성자는 rhs._ptr의 소유권을 공유합니다. rhs는 그대로 살아남고, 양쪽이 같은 객체를 가리킵니다.

4. 이동 생성자 — TSharedPtr(TSharedPtr&& rhs)

ServerCore/RefCounting.h : 42
TSharedPtr(TSharedPtr&& rhs) {
    _ptr = rhs._ptr;       // 1) 포인터를 훔쳐온다
    rhs._ptr = nullptr;     // 2) 원본은 비워둔다
}

의미

rvalue(곧 사라질 임시값) 로부터 새 객체를 만들 때 호출됩니다. 어차피 rhs는 곧 죽을 거라 refCount를 +1 하지 않고 그냥 포인터만 옮겨옵니다. 그래서 AddRef() 호출이 없습니다.

호출 예시

TSharedPtr<Player> a(new Player());      // ref=1
TSharedPtr<Player> b = std::move(a);        // 이동 → ref=1 그대로
                                            // a._ptr 은 nullptr

그림으로

왜 rhs._ptr = nullptr 가 중요한가

이걸 안 하면 rhs의 소멸자가 호출될 때 Release()가 돌면서 같은 객체의 refCount를 또 -1 시켜버립니다. 결과적으로 ref가 0이 돼서 아직 b가 살아있는데 객체가 delete 됩니다. use-after-free

5. 변환 생성자 — 다른 타입 SharedPtr로부터

ServerCore/RefCounting.h : 44-45
template<typename U>
TSharedPtr(const TSharedPtr<U>& rhs) {
    Set(static_cast<T>(rhs._ptr));
}

의미

TSharedPtr<Derived> 에서 TSharedPtr<Base> 로 자연스럽게 변환하기 위한 생성자입니다. 템플릿 인자 U는 T와 다른 타입(예: 자식 클래스)일 수 있다는 점이 포인트.

호출 예시

TSharedPtr<Player>     p(new Player());
TSharedPtr<RefCountable> base = p;   // Player → RefCountable
참고 — 코드의 오타 가능성 
static_cast<T>(rhs._ptr)     // ← 현재 코드
static_cast<T*>(rhs._ptr)    // ← 원래 의도(포인터로 캐스팅)
T로 캐스팅하면 객체 값으로 변환하려 들기 때문에 컴파일이 안 될 가능성이 높습니다. 실제 사용 시 T*로 고쳐야 정상 동작해요.

6. 복사 대입 연산자 — operator=(const TSharedPtr& rhs)

ServerCore/RefCounting.h : 51-58
TSharedPtr& operator=(const TSharedPtr& rhs) {
    if (_ptr != rhs._ptr) {     // 1) 자기 자신 / 동일 대입 가드
        Release();              // 2) 기존 자원 정리(ref--)
        Set(rhs._ptr);           // 3) 새 자원 대입(ref++)
    }
    return *this;             // 4) 연쇄 대입 a=b=c 지원
}

3단계 핵심 패턴

대입 연산자의 본질은 다음 세 가지를 이 순서로 하는 것입니다.

  1. 자기 대입 체크 — a = a; 일 때 Release하면 ref가 0으로 떨어져 객체가 사라질 수 있어 위험합니다. 그래서 같은 포인터면 그냥 건너뜀.
  2. 기존 것 정리 — 나는 이미 어떤 객체를 가리키고 있었으니, 그 객체의 refCount를 먼저 -1 해주고 시작.
  3. 새 것 받기 — 이제 rhs의 포인터를 받아 refCount +1.

그림으로

왜 return *this? C++의 =는 결과값이 있어야 a = b = c; 같은 연쇄 대입이 됩니다. 표준 라이브러리 관례를 따르기 위한 것.

7. 이동 대입 연산자 — operator=(TSharedPtr&& rhs)

ServerCore/RefCounting.h : 62-68
TSharedPtr& operator=(TSharedPtr&& rhs) {
    Release();          // 1) 내가 갖고 있던 것 정리
    _ptr = rhs._ptr;     // 2) 포인터 훔쳐오기 (AddRef 없음)
    rhs._ptr = nullptr;   // 3) 원본 무력화
    return *this;
}

복사 대입과의 차이

복사 대입이동 대입
인자 종류const T& lvalue T&& rvalue
refCount+1그대로(이전 소유자가 손을 뗌)
rhs 상태그대로nullptr로 비워짐
속도AddRef = atomic 연산 1회atomic 연산 없음 → 더 빠름

호출 예시

TSharedPtr<Player> a(new Player());      // ref=1
TSharedPtr<Player> b(new Player());      // ref=1
b = std::move(a);    // b가 들고 있던 Player는 ref=0 → delete
                     // b는 이제 a가 갖고 있던 Player를 가리킴, a는 nullptr
자기 대입 가드가 없네? — std::move(a) = a 같은 이상한 코드를 짜지 않는 한 자기 대입이 거의 발생하지 않아서 생략하는 게 일반적입니다. 필요하면 if (this != &rhs)를 추가해도 됩니다.

8. 그 외 연산자들 — 비교 · 역참조

비교 연산자

bool operator==(const TSharedPtr& rhs) const;
bool operator!=(const TSharedPtr& rhs) const;
bool operator<(const TSharedPtr& rhs) const;

모두 raw 포인터 주소 자체를 비교합니다. <는 std::map의 키로 쓸 수 있게 해주는 용도예요.

역참조 / 변환 연산자

T*        operator*();
T*        operator->();
          operator T*() const;

sp->Foo() 처럼 raw 포인터처럼 쓰게 해주는 핵심 연산자들. 암시적 T* 변환까지 정의해서 if (sp) ... 같이 쓸 수도 있습니다.

참고 — 표준 std::shared_ptr은 operator*()가 T&를 반환합니다. 이 프로젝트의 TSharedPtr은 T*를 반환해 조금 다른데, 단순화를 위한 선택으로 보입니다.

헷갈릴 때 떠올릴 3가지 질문

  1. "내가 지금 태어나는 거야, 살아있는 거야?" → 태어나면 생성자, 살아있으면 대입 연산자. 살아있으면 Release 먼저.
  2. "인자가 이름 있는 값(lvalue)이야, 곧 죽을 값(rvalue)이야?" → lvalue면 복사(AddRef +1), rvalue면 이동(포인터만 훔쳐옴).
  3. "내가 손을 떼면 rhs는 어떻게 돼?" → 복사면 rhs는 그대로, 이동이면 rhs._ptr = nullptr.
이 세 질문으로 5종 세트가 다 정리됩니다. 매번 처음부터 생각하려 하지 말고 "태어남/살아있음 → lvalue/rvalue" 순서로 분기치면 항상 맞는 함수가 골라져요.

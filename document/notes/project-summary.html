Lock-Based ServerCore 정리

RW SpinLock · DeadLock Profiler · ThreadManager 중심으로 본 프로젝트 한눈 정리

개요 솔루션 구조 Types & Macro TLS · ThreadManager CoreGlobal RW SpinLock DeadLock Profiler GameServer 데모 전체 동작 흐름 요약

1. 프로젝트 개요

이 프로젝트는 멀티스레드 IOCP 서버를 만들기 전 단계로, 락 시스템과 데드락 감지를 직접 구현해 보는 학습용 C++ 솔루션입니다. 핵심은 두 가지입니다.

① 직접 만든 RW SpinLock

한 개의 atomic<uint32> 안에 "쓰기 점유 스레드 ID"와 "읽기 카운트"를 비트로 패킹해서, 가벼운 Read/Write 락을 만들었습니다.

② DeadLock Profiler

스레드가 락을 잡는 순서를 그래프로 기록하고, DFS로 사이클을 찾아 데드락 가능성을 런타임에 잡아냅니다.

GameServer에서는 Player → Account와 Account → Player 순서로 락을 잡는 두 스레드를 일부러 돌려 프로파일러가 데드락을 감지하는 모습을 확인할 수 있게 되어 있습니다.

3. 공용 타입과 매크로

Types.h — 타입 별칭

ServerCore/Types.h
// 정수형, atomic, mutex 등을 짧은 이름으로 통일
using int8   = __int8;

template<typename T>
using Atomic    = std::atomic<T>;

이후 코드 전체에서 uint32, Atomic<T>, LockGuard 같이 한 줄로 짧게 쓸 수 있게 됩니다.

CoreMacro.h — 락 사용 매크로

ServerCore/CoreMacro.h
#define USE_MANY_LOCKS(count)  Lock _locks[count];
#define USE_LOCK               USE_MANY_LOCKS(1)

#define READ_LOCK_IDX(idx)
    ReadLockGuard  readLockGuard_##idx (_locks[idx], typeid(this).name());
#define WRITE_LOCK_IDX(idx)
    WriteLockGuard writeLockGuard_##idx(_locks[idx], typeid(this).name());

#define READ_LOCK               READ_LOCK_IDX(0)
#define WRITE_LOCK              WRITE_LOCK_IDX(0)

#define CRASH(cause) { uint32* p = nullptr; *p = 0xDEADBEEF; }
  • USE_LOCK — 클래스에 Lock _locks[1]; 멤버를 박아줍니다.
  • WRITE_LOCK — 한 줄로 WriteLockGuard를 만들고 typeid(this).name()으로 자동 이름표를 붙입니다.
  • CRASH — 의도적 NULL 역참조로 디버거에 멈춰서 콜스택을 보게 만듭니다.

이 매크로 덕분에 GameServer 쪽 코드가 WRITE_LOCK; 한 줄로 매우 간결하게 유지됩니다.

4. TLS & ThreadManager

CoreTLS — 스레드마다 고유 ID

ServerCore/CoreTLS.h / CoreTLS.cpp
extern thread_local uint32 LThreadId;
thread_local uint32 LThreadId = 0;

thread_local로 선언했기 때문에 스레드마다 별도의 LThreadId가 생깁니다. 이 값은 뒤에서 RW SpinLock이 "지금 누가 쓰기 락을 잡고 있나"를 구분하는 핵심 키로 쓰입니다.

ThreadManager — std::thread 생성/조인

ServerCore/ThreadManager.cpp
ThreadManager::ThreadManager() {
    InitTLS();                // 메인 스레드도 ID를 받음
}

void ThreadManager::Launch(function<void(void)> callback) {
    LockGuard guard(_lock);
    _threads.push_back(thread([=]() {
        InitTLS();           // 새 워커 스레드 ID 부여
        callback();           // 실제 작업 실행
        DestroyTLS();        // (현재는 비어 있음)
    }));
}

void ThreadManager::InitTLS() {
    static Atomic<uint32> SThreadId = 1;
    LThreadId = SThreadId.fetch_add(1);   // 1,2,3,... 순서대로
}
핵심 포인트 — SThreadId는 Atomic<uint32>여서 여러 스레드가 동시에 시작해도 안전하게 1씩 늘어난 번호를 받습니다. 각 스레드는 자기만의 LThreadId를 갖게 됩니다.

6. RW SpinLock — 이 프로젝트의 핵심 ①

비트 레이아웃

_lockFlag (uint32, atomic) ┌──────────────────────────┬──────────────────────────┐ │ 상위 16비트: WriteThreadId │ 하위 16비트: ReadCount │ └──────────────────────────┴──────────────────────────┘ WRITE_THREAD_MASK = 0xFFFF0000 READ_COUNT_MASK = 0x0000FFFF

한 개의 atomic<uint32>에 "지금 쓰기 락을 잡고 있는 스레드 ID"와 "현재 읽기 락 카운트"를 같이 저장합니다. 이렇게 묶어 두면 CAS 한 번으로 락 상태를 원자적으로 바꿀 수 있습니다.

규칙

상황허용?비고
같은 스레드의 W → W (재귀)OK _writeCount 증가만
같은 스레드의 W → R (소유 중 읽기)OK 읽기 카운트 +1
같은 스레드의 R → W금지 업그레이드 불가
다른 스레드의 W가 점유 중대기 스핀 → yield → 타임아웃 시 CRASH

WriteLock 코드

ServerCore/Lock.cpp
void Lock::WriteLock(const char* name) {
    // 1) 동일 스레드의 재귀 → 그냥 카운트만 증가
    const uint32 lockThreadId = (_lockFlag.load() & WRITE_THREAD_MASK) >> 16;
    if (LThreadId == lockThreadId) { _writeCount++; return; }

    // 2) 아무도 안 잡고 있을 때(EMPTY_FLAG)만 CAS 성공
    const uint32 desired = (LThreadId << 16) & WRITE_THREAD_MASK;
    while (true) {
        for (uint32 spin = 0; spin < MAX_SPIN_COUNT; spin++) {
            uint32 expected = EMPTY_FLAG;
            if (_lockFlag.compare_exchange_strong(expected, desired)) {
                _writeCount++;
                return;
            }
        }
        if (timeout) CRASH("LOCK_TIMEOUT");
        this_thread::yield();
    }
}

스핀 카운트(5000회) 동안만 빙빙 돌고, 그래도 못 얻으면 yield()로 OS에 양보합니다. 10초까지 못 얻으면 데드락으로 간주하고 의도적으로 CRASH합니다.

ReadLock 코드

void Lock::ReadLock(const char* name) {
    // 내가 이미 쓰기 락을 잡고 있다면 그냥 카운트만 +1
    const uint32 lockThreadId = (_lockFlag.load() & WRITE_THREAD_MASK) >> 16;
    if (LThreadId == lockThreadId) { _lockFlag.fetch_add(1); return; }

    // 쓰기 락이 없을 때만 ReadCount +1 시도
    while (true) {
        for (uint32 spin = 0; spin < MAX_SPIN_COUNT; spin++) {
            uint32 expected = (_lockFlag.load() & READ_COUNT_MASK); // W=0 인 상태
            if (_lockFlag.compare_exchange_strong(expected, expected + 1))
                return;
        }
        if (timeout) CRASH("LOCK_TIMEOUT");
        this_thread::yield();
    }
}

핵심 트릭은 expected = _lockFlag & READ_COUNT_MASK입니다. 이렇게 하면 "쓰기 비트가 0(누구도 쓰기 안 함)인 상황"에서만 CAS가 성공합니다.

LockGuard (RAII 래퍼)

class WriteLockGuard {
public:
    WriteLockGuard(Lock& lock, const char* name)
        : _lock(lock), _name(name) { _lock.WriteLock(name); }
    ~WriteLockGuard() { _lock.WriteUnlock(_name); }
};

생성 시 락을 잡고 소멸 시 자동으로 푸는 표준적인 RAII 패턴. WRITE_LOCK; 매크로 한 줄로 이걸 만듭니다.

7. DeadLock Profiler — 이 프로젝트의 핵심 ②

아이디어

데드락은 보통 락을 잡는 순서가 사이클을 이룰 때 발생합니다. 예를 들어 스레드 A는 L1 → L2, 스레드 B는 L2 → L1 순으로 잡으면, 서로가 서로를 기다리며 멈춥니다.

L1 ──▶ L2 ▲ │ └──────┘ ← 이런 화살표 사이클이 생기면 데드락 위험!

그래서 프로파일러는:

  • 락을 잡을 때마다 지금까지 들고 있던 락 → 새로 잡는 락의 간선을 그래프에 추가
  • 새 간선이 생기면 DFS로 사이클이 생겼는지 확인
  • 사이클이 발견되면 경로를 출력하고 즉시 CRASH

PushLock — 그래프에 간선 추가

ServerCore/DeadLockProfiler.cpp
void DeadLockProfiler::PushLock(const char* name) {
    LockGuard guard(_lock);

    // 1) 락 이름 ↔ 정수 ID 매핑(처음 보면 새 ID 발급)
    int32 lockId = ...;

    // 2) 지금까지 들고 있던 락이 있으면, prev → 현재 락 으로 간선 추가
    if (!_lockStack.empty()) {
        const int32 prevId = _lockStack.top();
        if (lockId != prevId) {
            auto& history = _lockHistory[prevId];
            if (history.insert(lockId).second) {
                CheckCycle();   // 새 간선이 생겼으니 검사
            }
        }
    }

    _lockStack.push(lockId);     // 내가 지금 들고 있는 락 스택에 추가
}

CheckCycle / DFS

void DeadLockProfiler::Dfs(int32 here) {
    if (_discoveredOrder[here] != -1) return;
    _discoveredOrder[here] = _discoveredCount++;

    for (int32 there : _lockHistory[here]) {
        if (_discoveredOrder[there] == -1) {       // 트리/전방 간선
            _parent[there] = here;
            Dfs(there);
            continue;
        }
        if (_discoveredOrder[here] < _discoveredOrder[there]) continue; // 교차 간선

        // here 보다 there 가 먼저 발견됐고, 아직 Dfs가 끝나지 않았다면
        // → there 는 here 의 조상 = 사이클(=back edge) !!
        if (!_finished[there]) {
            printf("%s -> %s\n", _idToName[here], _idToName[there]);
            // parent를 따라 사이클 경로를 거꾸로 인쇄
            CRASH("DEADLOCK_DETECTED");
        }
    }
    _finished[here] = true;
}
왜 이렇게 동작? DFS 도중 아직 끝나지 않은 노드로 거꾸로 가는 간선(=back edge)이 발견되면 그건 곧 그래프에 사이클이 있다는 뜻입니다. 데드락은 정확히 이 사이클을 따라 발생합니다.

PopLock — 락 해제 시

void DeadLockProfiler::PopLock(const char* name) {
    LockGuard guard(_lock);
    if (_lockStack.empty())          CRASH("MULTIPLE_UNLOCK");
    if (_lockStack.top() != _nameToId[name]) CRASH("INVALID_UNLOCK");
    _lockStack.pop();
}

스택의 top과 푸는 락이 다르면 락 해제 순서 위반이므로 즉시 CRASH합니다. LIFO 순서를 강제해 두면 그래프의 의미가 깨지지 않습니다.

이 PushLock/PopLock은 Lock::WriteLock/ReadLock 안에서 #if _DEBUG일 때만 호출됩니다. 디버그에서만 데드락을 잡고, 릴리즈에서는 비용 0이라는 뜻입니다.

8. GameServer — 데드락 시나리오 데모

두 매니저, 서로 반대 순서로 락 획득

GameServer/PlayerManager.cpp
void PlayerManager::PlayerThenAccount() {
    WRITE_LOCK;                // 1) PlayerManager 락
    GAccountManager.Lock();   // 2) 그 안에서 AccountManager 락
}
GameServer/AccountManager.cpp
void AccountManager::AccountThenPlayer() {
    WRITE_LOCK;                // 1) AccountManager 락
    GPlayerManager.Lock();    // 2) 그 안에서 PlayerManager 락
}

두 스레드를 동시에 돌리는 main

GameServer/GameServer.cpp
int main() {
    GThreadManager->Launch([=]() {
        while (true) {
            cout << "PlayerThenAccount" << endl;
            GPlayerManager.PlayerThenAccount();    // P → A 순서
            this_thread::sleep_for(100ms);
        }
    });

    GThreadManager->Launch([=]() {
        while (true) {
            cout << "AccountThenPlayer" << endl;
            GAccountManager.AccountThenPlayer();    // A → P 순서
            this_thread::sleep_for(100ms);
        }
    });

    GThreadManager->Join();
}
결과 — 두 스레드가 충분히 돌고 나면 DeadLockProfiler가 Player → Account와 Account → Player 두 간선이 사이클을 이룬다는 사실을 발견하고 DEADLOCK_DETECTED로 의도적 CRASH를 발생시킵니다. 데드락이 "조용히 굳어버리는 게 아니라" 명확하게 잡히도록 만든 것이 이 프로젝트의 핵심 성과입니다.

9. 전체 동작 흐름 한눈에

[프로그램 시작] │ ├─ GCoreGlobal 생성자 │ ├─ new ThreadManager() → 메인 스레드 InitTLS() │ └─ new DeadLockProfiler() │ ├─ main() │ ├─ GThreadManager->Launch(스레드1: PlayerThenAccount) │ │ └─ 새 std::thread 안에서 InitTLS() → callback() → DestroyTLS() │ ├─ GThreadManager->Launch(스레드2: AccountThenPlayer) │ └─ GThreadManager->Join() │ ├─ [스레드1] WRITE_LOCK(Player) → PushLock("Player") → 락 획득 │ WRITE_LOCK(Account) → PushLock("Account") → CheckCycle() │ ├─ [스레드2] WRITE_LOCK(Account) → PushLock("Account") → 락 획득 │ WRITE_LOCK(Player) → PushLock("Player") → CheckCycle() │ └─ [Profiler] Player→Account 와 Account→Player 두 간선이 사이클을 만들면 DFS가 back edge 발견 → CRASH("DEADLOCK_DETECTED")

10. 한 페이지 요약

지금까지 완성된 것

  • 공용 타입/매크로 정리 (Types.h, CoreMacro.h)
  • TLS 기반 스레드 ID (CoreTLS)
  • 스레드 생성/조인 관리 (ThreadManager)
  • 전역 초기화 진입점 (CoreGlobal)
  • RW SpinLock: 비트 패킹 + CAS + 재귀 지원
  • DeadLock Profiler: 락 의존 그래프 + DFS 사이클 검사
  • GameServer에서 실제 데드락이 감지되는 시나리오 데모

아직 안 들어간 것

  • IOCP, 소켓, 세션, 패킷 처리
  • DummyClient는 사실상 빈 상태
  • DestroyTLS()는 비어 있음
  • 워커 스레드 종료 조건 없음 (무한 루프)
  • 로그/에러 처리 시스템
이 프로젝트의 의미 — IOCP 서버를 만들기 전에, 여러 스레드가 안전하게 데이터에 접근하는 토대(락)와, 그 토대가 잘못 쓰였을 때 즉시 알려주는 안전망(데드락 감지)을 먼저 직접 만들어 본 단계입니다. 다음 단계에서 Winsock/IOCP를 얹으면 자연스럽게 멀티스레드 서버로 확장됩니다.

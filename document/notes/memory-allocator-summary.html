실제 메모리 확보와 해제를 담당한다. 현재는 내부적으로 ::malloc과 ::free를 사용한다.

Memory.h

ServerCore/Memory.h

템플릿 함수 xnew, xdelete를 제공해서 객체 생성과 소멸까지 처리한다.

핵심은 메모리 할당과 객체 생성이 서로 다른 단계라는 점이다. malloc은 바이트 공간만 만들고, 생성자는 호출하지 않는다.

2. Allocator.cpp: 원시 메모리 할당 계층

BaseAllocator는 가장 기본적인 메모리 할당기다. 지금은 단순히 C 런타임의 malloc/free를 감싸고 있지만, 나중에 메모리 풀, 디버그 태그, 누수 추적, 통계 수집 같은 기능을 넣기 좋은 지점이다. 

void* BaseAllocator::Alloc(int32 size)
{
    return ::malloc(size);
}

void BaseAllocator::Release(void* ptr)
{
    ::free(ptr);
}
함수 역할 중요 포인트
Alloc(int32 size) size 바이트만큼 원시 메모리를 확보한다. 생성자를 호출하지 않는다. 단지 주소만 돌려준다.
Release(void* ptr) 확보했던 원시 메모리를 해제한다. 소멸자를 호출하지 않는다. 객체 정리는 호출자가 먼저 해야 한다.

3. Memory.h: 객체 생성과 소멸 래퍼

Memory.h는 BaseAllocator 위에 있는 편의 계층이다. 사용자는 xnew<Type>(...)와 xdelete(ptr)만 호출하면 된다.

xnew

template<typename Type, typename... Args>
Type* xnew(Args&&... args)
{
    Type* memory = static_cast<Type*>(xalloc(sizeof(Type)));

    // placement new
    new(memory) Type(std::forward<Args>(args)...);
    return memory;
}
  1. sizeof(Type)으로 객체 하나가 들어갈 메모리 크기를 구한다.
  2. xalloc으로 원시 메모리를 확보한다.
  3. placement new로 이미 확보한 메모리 위에 객체를 생성한다.
  4. std::forward로 생성자 인자를 최대한 원래 형태 그대로 전달한다.

xdelete

template<typename Type>
void xdelete(Type* obj)
{
    obj->~Type();
    xrelease(obj);
}
  1. obj->~Type()로 소멸자를 직접 호출한다.
  2. xrelease로 객체가 놓여 있던 메모리를 해제한다.
placement new를 사용했기 때문에 일반 delete obj를 호출하면 안 된다. 생성과 해제를 직접 나눴으므로, 소멸자 호출과 메모리 해제도 직접 순서대로 처리해야 한다.

4. 전체 호출 흐름

GameServer.cpp에서 xnew<Knight>(100)을 호출하면 실제로는 아래 순서로 진행된다.

GameServer.cpp Knight* k = xnew<Knight>(100); | v Memory.h::xnew<Knight, int>(100) xalloc(sizeof(Knight)) | v CoreMacro.h xalloc(size) -> BaseAllocator::Alloc(size) | v Allocator.cpp ::malloc(size) | v Memory.h new(memory) Knight(100) | v Knight 객체 생성 완료

삭제는 반대 방향으로 생각하면 된다.

GameServer.cpp xdelete(k); | v Memory.h::xdelete<Knight>(k) k->~Knight() | v CoreMacro.h xrelease(k) -> BaseAllocator::Release(k) | v Allocator.cpp ::free(k)

5. 일반 new/delete와 비교

구분 일반 new/delete 현재 xnew/xdelete
메모리 확보 operator new가 처리 BaseAllocator::Alloc이 처리
생성자 호출 new Type(...)가 자동 호출 placement new로 직접 호출
소멸자 호출 delete ptr가 자동 호출 obj->~Type()로 직접 호출
메모리 해제 operator delete가 처리 BaseAllocator::Release가 처리
확장성 전역 new/delete 오버로딩 또는 클래스별 오버로딩 필요 Allocator 내부만 바꾸면 전체 정책을 바꾸기 쉽다

6. 주의점과 개선 포인트

주의nullptr 처리

현재 xdelete(nullptr)를 호출하면 obj->~Type()에서 문제가 생길 수 있다. 일반 delete nullptr처럼 안전하게 만들려면 null 체크를 추가하는 편이 좋다. 

if (obj == nullptr)
    return;

주의배열은 별도 처리 필요

지금 xnew/xdelete는 객체 하나 기준이다. 배열을 지원하려면 원소 개수 저장, 반복 생성자 호출, 역순 소멸자 호출 같은 추가 처리가 필요하다.

설계Debug/Release 분기

CoreMacro.h에는 Debug/Release 분기가 있지만 현재 둘 다 같은 allocator를 사용한다. 나중에 Debug에서 파일명, 라인 번호, 할당 크기 등을 기록하는 구조로 확장할 수 있다.

빌드라이브러리 갱신

Allocator.cpp가 ServerCore.lib에 포함되어야 BaseAllocator::Alloc/Release 링크 에러가 나지 않는다. GameServer는 ServerCore를 먼저 빌드한 뒤 링크해야 한다.

중요: malloc/free와 생성자/소멸자는 자동으로 연결되지 않는다. 그래서 xnew에서는 placement new가 필요하고, xdelete에서는 명시적 소멸자 호출이 필요하다.

7. 한 줄 요약

Allocator.cpp는 메모리 공간을 빌리고 반납하는 하위 계층이고, Memory.h는 그 공간 위에서 객체의 생명주기까지 관리하는 상위 계층이다. 이 구조를 만들면 나중에 서버 전용 메모리 관리자, 메모리 풀, 디버그 추적 기능을 자연스럽게 끼워 넣을 수 있다.

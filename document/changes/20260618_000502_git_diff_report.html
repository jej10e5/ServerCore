TCP 접속 실행 흐름

`GameServer`와 `DummyClient` 모두 WinSock을 초기화한 뒤 IPv4 TCP 소켓을 만들고, 같은 포트 7777을 기준으로 서로 다른 역할을 수행합니다. 서버는 주소 바인딩, listen, accept를 담당하고 클라이언트는 루프백 주소로 connect를 수행합니다.

코드 역할: 서버 실행 프로세스에서 WinSock DLL을 초기화하고 IPv4/TCP listen 소켓을 생성합니다. 이후 `bind`, `listen`, `accept`의 기반이 되는 대표 소켓입니다.
SOCKADDR_IN serverAddr;
::memset(&serverAddr, 0, sizeof(serverAddr));
serverAddr.sin_family = AF_INET;
serverAddr.sin_addr.s_addr = ::htonl(INADDR_ANY);
serverAddr.sin_port = ::htons(7777);

if (::bind(listenSocket, (SOCKADDR*)&serverAddr, sizeof(serverAddr)) == SOCKET_ERROR)
    return 0;

if (::listen(listenSocket, 10) == SOCKET_ERROR)
    return 0;
GameServer/GameServer.cpp
코드 역할: 서버가 모든 로컬 인터페이스에서 7777 포트를 열고 최대 backlog 10으로 접속 대기를 시작합니다. 이 변경으로 `GameServer`는 단순 샘플 실행기가 아니라 외부 접속을 받을 수 있는 프로세스가 됩니다.
while (true)
{
    SOCKADDR_IN clientAddr;
    ::memset(&clientAddr, 0, sizeof(clientAddr));
    int32 addrLen = sizeof(clientAddr);

    SOCKET clientSocket = ::accept(listenSocket, (SOCKADDR*)&clientAddr, &addrLen);
    if (clientSocket == INVALID_SOCKET)
        return 0;

    char ipAddress[16];
    ::inet_ntop(AF_INET, &clientAddr.sin_addr, ipAddress, sizeof(ipAddress));
    cout << "Client Connected! IP = " << ipAddress << endl;
}

DummyClient/DummyClient.cpp
코드 역할: 더미 클라이언트가 루프백 주소와 7777 포트를 목적지로 설정한 뒤 서버에 접속합니다. 서버와 같은 포트/프로토콜을 사용하므로 로컬 머신에서 접속 시나리오를 바로 검증할 수 있습니다.

협력 방식: 서버의 `INADDR_ANY:7777` listen 소켓과 클라이언트의 `127.0.0.1:7777` connect 호출이 맞물려 최소 TCP 핸드셰이크 테스트를 구성합니다. 서버는 연결된 클라이언트별 소켓을 얻지만 아직 해당 소켓으로 send/recv를 수행하지는 않습니다.

확인 포인트

  • `while (true)` 이후에 정상 종료 경로가 없어 `closesocket`과 `WSACleanup`은 현재 실행 흐름에서 도달하기 어렵습니다.
  • `GameServer`는 `accept`가 실패하면 즉시 프로세스를 종료합니다. 장기 실행 서버라면 오류 종류별 재시도/종료 정책이 필요합니다.
  • `DummyClient`의 `int32 num = 0x12345678;`은 엔디언 설명용으로 보이지만 실제 사용되지 않습니다.
  • 현재 변경은 접속 확인까지이며, 패킷 송수신/세션 관리/비동기 I/O는 아직 구현되지 않았습니다.

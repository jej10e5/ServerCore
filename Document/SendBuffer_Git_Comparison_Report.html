메모리와 수명 흐름

1. Open — TLS 청크가 없으면 전역 풀에서 꺼내고, 남은 공간이 요청 크기보다 작아도 새 청크로 교체한다.
2. Reserve — 청크의 _usedSize 위치를 시작 주소로 하는 SendBuffer를 ObjectPool에서 만든다.
3. Write / Close — 호출자가 데이터를 기록한 뒤 실제 기록 길이를 확정한다. 청크는 그 길이만큼 _usedSize를 전진시킨다.
4. Async Send — 세션의 SendEvent가 SendBufferRef를 보관하고, 버퍼의 _owner가 청크를 계속 살려 둔다.
5. Recycle — TLS와 모든 송신 참조가 사라지면 커스텀 deleter가 청크를 전역 관리자 목록에 반환한다.

역할 구분
청크의 _usedSize는 큰 저장소가 전체적으로 어디까지 사용됐는지 관리한다. 각 SendBuffer는 그 안의 특정 패킷 구간을 나타내는 뷰이며, _owner를 통해 원본 청크를 붙잡는다. 따라서 청크 풀링과 SendBuffer 오브젝트 풀링은 중복이 아니라 서로 다른 메모리와 수명을 최적화한다.
shared_ptr 제어 블록
현재 ObjectPool<SendBuffer>는 SendBuffer 본체 메모리를 재사용한다. 하지만 raw pointer와 커스텀 deleter로 생성한 shared_ptr의 제어 블록은 일반적으로 별도 동적 할당되므로 완전한 무할당 구조는 아니다.

Close()가 끝난 패킷 뒤로 _usedSize가 이동하고, 다음 Open()은 화살표 위치부터 시작한다. 구간 너비는 이해를 위한 표현이며 실제 바이트 비율과는 다르다.

SendBuffer는 각 패킷에 붙는 택배 송장

하나의 큰 상자 안에는 여러 패킷이 있으므로 각 패킷의 위치와 크기를 설명하는 관리 정보가 필요하다.

왜 shared_ptr인가?
택배 운송이 끝나기 전에 송장과 상자를 버리면 안 되는 것처럼, 비동기 WSASend가 끝날 때까지 shared_ptr<SendBuffer>가 송장을 보관하고 송장의 _owner가 큰 상자를 붙잡는다. 송신 완료 후 모든 참조가 사라지면 송장은 ObjectPool로, 상자는 SendBufferManager의 청크 풀로 돌아간다.

현재 로컬에 추가된 방어 코드

잔여 공간 재검사 — SendBufferChunk::Open()에 if (allocSize > FreeSize()) return nullptr;가 추가됐다. Manager에서도 먼저 검사하지만 Chunk 직접 호출이나 향후 코드 변경을 방어하는 이중 검증이다.
Reset 의도 명시 — Reset()은 byte 배열을 0으로 지우지 않고 _open = false, _usedSize = 0만 수행한다. 이후 기록 데이터가 기존 내용을 덮고 실제 WriteSize()만 송신한다.

기대 효과

  • 동적 할당 감소: 매 송신마다 byte vector를 새로 확보하는 비용을 줄인다.
  • 지역성 개선: 한 스레드가 생성한 여러 패킷이 같은 연속 청크에 배치된다.
  • 비동기 수명 보장: SendBuffer가 owner 청크를 공유 소유하므로 WSASend 완료 전 메모리가 재사용되지 않는다.
  • 경합 축소: 평상시 할당은 TLS 청크에서 처리하며, 청크 교체/반환 때만 전역 락을 사용한다.

확인된 주의점과 위험

높음 — 전역 종료 순서 위험
delete GSendBufferManager 중 내부 vector의 청크 참조가 파괴되면 커스텀 deleter PushGlobal이 다시 같은 관리자에 청크를 넣는다. 객체가 소멸 중인 상태에서 멤버와 락에 재진입하므로 종료 시 미정의 동작 또는 누수/크래시 가능성이 있다. 종료 전 반환 경로를 끄거나, 풀의 raw ownership/명시적 destroy 정책을 분리하는 편이 안전하다.
중간 — Close가 API의 필수 계약
Close()를 누락하면 WriteSize가 0이고 청크가 열린 채 남아 다음 Open()에서 assert가 발생한다. 두 번 호출하면 보통 assert되지만, 사이에 다른 버퍼가 열렸다면 이전 버퍼의 중복 Close가 새 버퍼 상태를 잘못 닫을 여지도 있다. SendBuffer 자체에 closed 상태를 두는 방어가 없다.
중간 — 요청 크기 상한
한 번의 Open(size)는 6,000바이트 이하여야 한다. 기존 생성자는 임의의 크기를 받을 수 있었으므로 향후 6,000바이트 초과 패킷은 assert로 종료된다.
중간 — 기능 변경 혼입
GameSession::OnRecv의 Broadcast 반복이 5회에서 1회로 바뀌었다. SendBuffer 구조 전환과 별개의 동작 변화이므로 의도 여부를 확인해야 한다.
낮음 — 디버그 출력과 형식
매 Open마다 cout << "FREE ..."가 실행되어 고빈도 송신 경로 성능과 로그에 영향을 준다. 또한 git diff --check에서 trailing whitespace 1건과 파일 끝 빈 줄 경고가 확인됐다.

새 호출 규칙

SendBufferRef sendBuffer = GSendBufferManager->Open(allocSize);
::memcpy(sendBuffer->Buffer(), source, writeSize);
sendBuffer->Close(writeSize);   // writeSize <= allocSize, 정확히 한 번
Send(sendBuffer);

동시에 같은 스레드에서 두 버퍼를 Open할 수 없다. 첫 버퍼를 Close한 뒤 다음 버퍼를 Open해야 한다.

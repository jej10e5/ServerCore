PacketSession 로컬 변경 분석

Git HEAD 342861a와 현재 작업 트리를 비교해, TCP 수신 바이트를 완성된 패킷 단위로 분리하는 PacketSession의 구조와 적용 범위를 정리한다.

한눈에 보는 변화

기존 Session::OnRecv() 구현자는 수신된 바이트의 경계를 직접 해석해야 했다. 이제 콘텐츠 세션은 PacketSession을 상속하고 OnRecvPacket()에서 완성된 패킷만 처리한다.

수신 처리 흐름

  1. Session::ProcessRecv()가 새 바이트를 RecvBuffer에 기록한다.
  2. PacketSession::OnRecv(buffer, len)이 아직 처리하지 않은 길이를 계산한다.
  3. 남은 데이터가 4바이트 헤더보다 작으면 다음 수신을 기다린다.
  4. 남은 데이터가 header.size보다 작으면 불완전한 패킷이므로 대기한다.
  5. 완성된 패킷이면 OnRecvPacket()을 호출하고 다음 패킷을 계속 검사한다.
  6. 처리한 바이트 수만 반환해 나머지 조각을 RecvBuffer에 보존한다.
int32 PacketSession::OnRecv(BYTE* buffer, int32 len)
{
    int32 processLen = 0;
    while (true)
    {
        int32 dataSize = len - processLen;
        if (dataSize < sizeof(PacketHeader))
            break;

        PacketHeader header =
            *(reinterpret_cast<PacketHeader*>(&buffer[processLen]));
        if (dataSize < header.size)
            break;

        OnRecvPacket(&buffer[processLen], header.size);
        processLen += header.size;
    }
    return processLen;
}

함께 반영된 Session 수명·동시성 변경

  • Send()는 연결 상태를 먼저 확인하고, 잠금 구간 밖에서 RegisterSend()를 호출한다.
  • Disconnect()의 콘텐츠 콜백과 세션 해제는 ProcessDisconnect() 완료 시점으로 이동했다.
  • 연결 종료 콜백이 실제 IOCP disconnect 완료 흐름에 맞춰 실행된다.

검토 시 주의할 점

필수 방어 로직: 현재 코드는 header.size가 0이거나 헤더보다 작은 값을 거부하지 않는다. 이 경우 processLen이 증가하지 않아 무한 반복하거나 잘못된 경계로 파싱할 수 있다.

권장 검사: header.size >= sizeof(PacketHeader)와 프로토콜 최대 패킷 크기를 OnRecvPacket() 호출 전에 확인해야 한다.

콜백 반환값: OnRecvPacket()의 반환값은 현재 무시된다. 항상 전체 패킷을 소비하는 계약이면 반환형을 void로 단순화하거나 실패를 표현할 계약을 명확히 하는 편이 좋다.

payload 위치: 호출부의 &buffer[4] 대신 buffer + sizeof(PacketHeader)를 사용하면 헤더 구조 변경에 안전하다.

현재 적용 결과

로컬 변경은 TCP의 패킷 합쳐짐과 잘림을 공통 계층에서 처리하는 기반을 추가했다. 콘텐츠 코드는 수신 버퍼 관리에서 분리되어 프로토콜 ID와 payload 처리에 집중할 수 있다. 다만 외부 입력을 신뢰하기 전에 패킷 최소·최대 크기 검증을 추가해야 운영 환경에서 안전하다.

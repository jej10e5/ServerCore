IOCP owner 기반 Dispatch 흐름

Service 추가 이후의 로컬 구현을 기준으로 정리한다. 현재 코드는 CreateIoCompletionPort의 completion key가 아니라 IocpEvent::owner를 보고 실제 Dispatch 대상 객체를 찾는다. 그래서 accept 완료는 Listener::Dispatch로 가고, recv/send 완료는 각 이벤트의 owner를 Session으로 넣었을 때 Session::Dispatch로 간다.

핵심 결론

IOCP에 socket을 등록하는 일과, 완료 이벤트를 어떤 객체가 처리할지 정하는 일은 현재 코드에서 분리되어 있다. Service::CreateSession은 session socket을 IOCP에 연결하지만, 그 자체만으로 Session::Dispatch가 호출되지는 않는다. 실제 호출 대상은 GetQueuedCompletionStatus가 돌려준 IocpEvent 안의 owner가 결정한다.

현재 로컬 코드의 라우팅 기준

AcceptEvent owner = Listener session = 새 Session
AcceptEx 완료
IocpCore::Dispatch iocpEvent->owner 확인 owner->Dispatch(...)
owner가 Listener
Listener::Dispatch AcceptEvent 처리 ProcessAccept 호출

IocpCore::Register

현재 Register는 객체의 socket handle을 IOCP handle에 연결한다. 하지만 completion key에는 객체 포인터를 넣지 않고 0을 넣는다. 따라서 완료 통지를 꺼낼 때 key만 봐서는 Listener인지 Session인지 알 수 없다. 

bool IocpCore::Register(IocpObjectRef iocpObject)
{
    return ::CreateIoCompletionPort(
        iocpObject->GetHandle(),
        _iocpHandle,
        /*key*/0,
        0);
}

IocpCore::Dispatch

completion key 대신 OVERLAPPED로 돌아온 이벤트 객체를 IocpEvent*로 해석한다. 그리고 그 이벤트의 owner를 꺼내 다형성으로 Dispatch를 호출한다. 

bool IocpCore::Dispatch(uint32 timeoutMs)
{
    DWORD numOfBytes = 0;
    ULONG_PTR key = 0;
    IocpEvent* iocpEvent = nullptr;

    if (::GetQueuedCompletionStatus(
        _iocpHandle,
        OUT &numOfBytes,
        OUT &key,
        OUT reinterpret_cast<LPOVERLAPPED*>(&iocpEvent),
        timeoutMs))
    {
        IocpObjectRef iocpObject = iocpEvent->owner;
        iocpObject->Dispatch(iocpEvent, numOfBytes);
    }

    return true;
}

왜 Accept는 Listener로 가나?

accept 작업에 사용하는 이벤트는 AcceptEvent다. Listener::StartAccept에서 이 이벤트의 owner를 shared_from_this(), 즉 Listener 자신으로 지정한다. 그래서 AcceptEx가 완료되면 IOCP에서 되돌아온 이벤트의 owner가 Listener이고, 결과적으로 Listener::Dispatch가 실행된다. 

const int32 acceptCount = _service->GetMaxSessionCount();
for (int32 i = 0; i < acceptCount; i++)
{
    AcceptEvent* acceptEvent = xnew<AcceptEvent>();
    acceptEvent->owner = shared_from_this();
    _acceptEvents.push_back(acceptEvent);
    RegisterAccept(acceptEvent);
}

RegisterAccept

RegisterAccept는 다음 접속을 받기 위해 새 Session을 만들고, 그 session의 client socket을 AcceptEx에 넘긴다. 여기서 acceptEvent->session은 accept가 끝났을 때 어떤 Session이 연결되었는지 알기 위한 저장 공간이다. Dispatch 대상은 여전히 acceptEvent->owner인 Listener다. 

void Listener::RegisterAccept(AcceptEvent* acceptEvent)
{
    SessionRef session = _service->CreateSession();

    acceptEvent->Init();
    acceptEvent->session = session;

    DWORD bytesReceived = 0;
    SocketUtils::AcceptEx(
        _socket,
        session->GetSocket(),
        session->_recvBuffer,
        0,
        sizeof(SOCKADDR_IN) + 16,
        sizeof(SOCKADDR_IN) + 16,
        OUT &bytesReceived,
        static_cast<LPOVERLAPPED>(acceptEvent));
}

Service::CreateSession

Service는 factory로 실제 Session 타입을 만든다. GameServer에서는 MakeShared<GameSession>을 넘기므로, 여기서 생성되는 객체는 Session을 상속한 GameSession이다. 생성 직후 session socket도 IOCP에 등록된다. 

SessionRef Service::CreateSession()
{
    SessionRef session = _sessionFactory();
    if (_iocpCore->Register(session) == false)
        return nullptr;

    return session;
}

이 등록은 중요하다. 나중에 이 session socket에 대해 WSARecv나 WSASend 같은 overlapped I/O를 걸면, 그 완료 통지가 같은 IOCP queue로 들어온다. 다만 어떤 객체의 Dispatch를 부를지는 그때 넘긴 이벤트의 owner가 결정한다.

Session::Dispatch가 실행되는 조건

Session::Dispatch가 실행되려면 recv/send/connect 같은 Session 소유 작업의 이벤트 owner를 Session으로 세팅해야 한다. 아직 현재 Session::Dispatch는 TODO 상태이므로, 이후 recv/send 구현에서 다음과 같은 구조가 들어오게 된다. 

void Session::RegisterRecv()
{
    _recvEvent.Init();
    _recvEvent.owner = shared_from_this();

    WSABUF wsaBuf;
    wsaBuf.buf = _recvBuffer;
    wsaBuf.len = sizeof(_recvBuffer);

    DWORD numOfBytes = 0;
    DWORD flags = 0;
    ::WSARecv(
        _socket,
        &wsaBuf,
        1,
        OUT &numOfBytes,
        OUT &flags,
        static_cast<LPWSAOVERLAPPED>(&_recvEvent),
        nullptr);
}

Accept 이후 Recv가 추가되면 생기는 흐름

ProcessAccept -> session->RegisterRecv -> RecvEvent owner = Session -> IOCP queue -> IocpCore::Dispatch -> Session::Dispatch

한 줄로 정리

  • Register(listener): listen socket의 완료 통지를 IOCP로 보내게 연결한다.
  • acceptEvent->owner = Listener: accept 완료 처리자는 Listener라고 표시한다.
  • Register(session): session socket의 완료 통지를 IOCP로 보내게 연결한다.
  • recvEvent->owner = Session: recv 완료 처리자는 Session이라고 표시한다.
  • IocpCore::Dispatch: 완료된 이벤트의 owner를 꺼내 그 객체의 Dispatch를 호출한다.

읽을 때 주의할 점

예전 IOCP 설명에서는 completion key에 IocpObject*를 넣는 방식으로 설명할 수 있다. 하지만 지금 로컬 코드는 key에 0을 넣고, IocpEvent::owner를 사용한다. 두 방식 모두 가능하지만, 한 코드 안에서는 기준을 섞지 말고 일관되게 따라가야 한다.

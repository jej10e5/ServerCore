Overlapped I/O

관련 커밋: 20680b2 Overlapped 모델 이벤트 기반, cf77746 Overlapped 콜백 기반

개념 한눈에 보기

Overlapped I/O는 작업이 가능해질 때까지 기다렸다가 호출하는 모델이 아니라, 먼저 비동기 작업을 요청하고 완료를 통지받는 모델입니다. WSA_IO_PENDING은 실패가 아니라 “요청이 접수됐고 나중에 완료된다”는 의미입니다. 완료 통지는 이벤트 객체, 완료 루틴 콜백, 이후 단계의 IOCP 같은 방식으로 받을 수 있습니다.

Proactor 스타일 처리 흐름

WSARecv 요청 → WSA_IO_PENDING → OS가 I/O 진행 → 완료 통지 → 결과 처리

무엇을 배웠나

Overlapped I/O는 WSARecv, WSASend 같은 비동기 함수를 먼저 걸어두고, 완료를 나중에 통지받는 모델입니다. 이전 단계의 select, WSAEventSelect가 “가능한 시점”을 보고 나서 호출하는 Reactor 성격이라면, Overlapped I/O는 작업을 미리 요청하고 완료를 받는 Proactor 성격에 더 가깝습니다.

세션 구조

const int32 BUFSIZE = 1000;

struct Session
{
    SOCKET socket = INVALID_SOCKET;
    char recvBuffer[BUFSIZE] = {};
    int32 recvBytes = 0;
    int32 sendBytes = 0;
    WSAOVERLAPPED overlapped = {};
};

이벤트 기반 완료

WSAEVENT wsaEvent = ::WSACreateEvent();
session.overlapped.hEvent = wsaEvent;

WSABUF wsaBuf;
wsaBuf.buf = session.recvBuffer;
wsaBuf.len = BUFSIZE;

DWORD recvLen = 0;
DWORD flags = 0;
::memset(&session.overlapped, 0, sizeof(session.overlapped));
session.overlapped.hEvent = wsaEvent;

if (::WSARecv(clientSocket, &wsaBuf, 1, &recvLen, &flags, &session.overlapped, nullptr) == SOCKET_ERROR)
{
    if (::WSAGetLastError() == WSA_IO_PENDING)
    {
        ::WSAWaitForMultipleEvents(1, &wsaEvent, TRUE, WSA_INFINITE, FALSE);

        if (::WSAGetOverlappedResult(session.socket, &session.overlapped, &recvLen, FALSE, &flags) == FALSE)
            break;
    }
}

WSA_IO_PENDING은 실패가 아니라 “요청은 접수됐고 아직 완료되지 않았다”는 의미입니다. 이벤트 기반 방식에서는 완료 시 이벤트가 signaled 되고, WSAGetOverlappedResult로 실제 전송 바이트 수를 확인합니다.

콜백 기반 완료 루틴

struct Session
{
    WSAOVERLAPPED overlapped = {};
    SOCKET socket = INVALID_SOCKET;
    char recvBuffer[BUFSIZE] = {};
    int32 recvBytes = 0;
    int32 sendBytes = 0;
};

void CALLBACK RecvCallback(
    DWORD error,
    DWORD recvLen,
    LPWSAOVERLAPPED overlapped,
    DWORD flags)
{
    cout << "Data Recv Len Callback = " << recvLen << endl;

    Session* session = (Session*)overlapped;
}

최신 커밋에서는 WSAOVERLAPPED를 Session의 첫 번째 멤버로 옮겼습니다. 이렇게 하면 콜백에서 받은 LPWSAOVERLAPPED 주소가 곧 Session 객체의 시작 주소가 되므로, 실습 코드처럼 캐스팅해 세션을 복원할 수 있습니다. 실전 코드에서는 보통 CONTAINING_RECORD 같은 의도를 드러내는 방식이나 별도 컨텍스트 구조를 사용해 더 안전하게 표현합니다.

Alertable Wait

if (::WSARecv(clientSocket, &wsaBuf, 1, &recvLen, &flags, &session.overlapped, RecvCallback) == SOCKET_ERROR)
{
    if (::WSAGetLastError() == WSA_IO_PENDING)
    {
        ::SleepEx(INFINITE, TRUE);
    }
}

완료 루틴 콜백은 요청한 스레드가 alertable wait 상태에 들어가야 호출됩니다. SleepEx, WaitForSingleObjectEx, WaitForMultipleObjectsEx, WSAWaitForMultipleEvents의 alertable 옵션이 여기에 해당합니다.

모델 정리

  • select, WSAEventSelect: 준비 상태를 감지한 뒤 recv/send를 호출합니다.
  • Overlapped 이벤트 기반: 비동기 작업을 요청하고 이벤트로 완료를 확인합니다.
  • Overlapped 콜백 기반: 비동기 작업 완료 시 운영체제가 완료 루틴을 호출합니다.

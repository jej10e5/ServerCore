IOCP Accept 흐름 Overview

2026-07-08 학습 기록. 이번 단계에서는 Overlapped I/O에서 한 단계 더 나아가, 완료 통지를 IOCP 큐로 모으고 worker thread가 꺼내 처리하는 전체 흐름을 정리한다. 핵심은 "작업을 먼저 걸어두고, 완료되면 IOCP가 알려주고, 완료 이벤트에 담긴 타입으로 실제 처리 함수를 찾아간다"는 구조다.

한 문장으로 보기

Listener가 listen socket을 만들고 IOCP에 등록한 뒤 AcceptEx를 미리 걸어둔다. 클라이언트가 접속하면 Windows 커널이 accept 완료 이벤트를 IOCP 큐에 넣고, worker thread가 GetQueuedCompletionStatus로 꺼낸 다음 completion key에 들어 있던 IocpObject의 Dispatch를 호출한다.

전체 accept 처리 흐름

GameServer main -> Listener::StartAccept -> GIocpCore.Register -> bind / listen -> RegisterAccept -> AcceptEx pending -> IOCP queue -> IocpCore::Dispatch -> Listener::Dispatch -> ProcessAccept

초기화 단계

프로그램이 시작되면 전역 객체 초기화 과정에서 SocketUtils::Init이 호출된다. 여기서 WSAStartup으로 Winsock을 초기화하고, WSAIoctl을 통해 ConnectEx, DisconnectEx, AcceptEx 함수 포인터를 얻는다. 이 함수들은 일반 함수처럼 바로 링크되는 API가 아니라, 실행 중에 소켓을 통해 주소를 얻어와 사용하는 확장 함수다. 

void SocketUtils::Init()
{
    WSADATA wsaData;
    ASSERT_CRASH(::WSAStartup(MAKEWORD(2, 2), OUT &wsaData) == 0);

    SOCKET dummySocket = CreateSocket();
    ASSERT_CRASH(BindWindowsFunction(dummySocket, WSAID_CONNECTEX, reinterpret_cast<LPVOID*>(&ConnectEx)));
    ASSERT_CRASH(BindWindowsFunction(dummySocket, WSAID_DISCONNECTEX, reinterpret_cast<LPVOID*>(&DisconnectEx)));
    ASSERT_CRASH(BindWindowsFunction(dummySocket, WSAID_ACCEPTEX, reinterpret_cast<LPVOID*>(&AcceptEx)));
    Close(dummySocket);
}

IOCP 등록의 의미

IocpCore는 내부에 IOCP handle 하나를 가지고 있다. 처음 생성할 때는 CreateIoCompletionPort(INVALID_HANDLE_VALUE, ...)로 "완료 큐" 자체를 만든다. 이후 Register를 호출하면 특정 socket handle을 이 완료 큐에 연결한다.

중요한 부분은 세 번째 인자인 completion key다. 현재 코드는 completion key에 IocpObject*를 넣는다. 그래서 나중에 완료 통지를 꺼낼 때 "이 완료 이벤트는 어떤 객체에게 보내야 하는가?"를 바로 알 수 있다. 

bool IocpCore::Register(IocpObject* iocpObject)
{
    return ::CreateIoCompletionPort(
        iocpObject->GetHandle(),
        _iocpHandle,
        reinterpret_cast<ULONG_PTR>(iocpObject),
        0);
}

Listener::StartAccept

Listener는 서버의 입구 역할을 한다. listen socket을 만들고, IOCP에 listen socket을 등록한 뒤, 주소 재사용 옵션과 linger 옵션을 설정한다. 그 다음 bind로 IP/PORT를 소켓에 묶고 listen으로 접속 대기 상태에 들어간다.

여기까지는 "서버 문을 열었다"에 가깝고, 아직 IOCP 이벤트가 발생할 작업은 없다. IOCP에 완료 이벤트가 들어오려면 반드시 AcceptEx, WSARecv, WSASend 같은 Overlapped 작업을 먼저 걸어야 한다. 

bool Listener::StartAccept(NetAddress netAddress)
{
    _socket = SocketUtils::CreateSocket();
    if (_socket == INVALID_SOCKET)
        return false;

    if (GIocpCore.Register(this) == false)
        return false;

    if (SocketUtils::SetReuseAddress(_socket, true) == false)
        return false;

    if (SocketUtils::SetLinger(_socket, 0, 0) == false)
        return false;

    if (SocketUtils::Bind(_socket, netAddress) == false)
        return false;

    if (SocketUtils::Listen(_socket) == false)
        return false;

    AcceptEvent* acceptEvent = xnew<AcceptEvent>();
    _acceptEvents.push_back(acceptEvent);
    RegisterAccept(acceptEvent);

    return true;
}

디버깅 체크 포인트

IocpCore::Dispatch의 iocpObject->Dispatch(...)까지 내려오지 않는다면, 대부분 그 전 단계에서 완료 이벤트가 만들어지지 않은 것이다. bind, listen, AcceptEx가 성공했는지와 AcceptEx 실패 시 에러가 WSA_IO_PENDING인지 먼저 확인한다. WSA_IO_PENDING은 실패가 아니라 "작업이 정상 등록되었고 아직 완료되지 않았다"는 뜻이다.

AcceptEvent와 Session

AcceptEvent는 IocpEvent를 상속하고, 내부에 새로 연결될 Session*을 들고 있다. IocpEvent는 다시 OVERLAPPED를 상속한다. 따라서 AcceptEx에 AcceptEvent*를 LPOVERLAPPED로 넘길 수 있고, 완료 후 GetQueuedCompletionStatus에서 다시 IocpEvent*로 받을 수 있다.

class IocpEvent : public OVERLAPPED
{
public:
    IocpEvent(EventType type);
    void Init();
    EventType GetType() { return _type; }

protected:
    EventType _type;
};

class AcceptEvent : public IocpEvent
{
public:
    AcceptEvent() : IocpEvent(EventType::Accept) {}

    void SetSession(Session* session) { _session = session; }
    Session* GetSession() { return _session; }

private:
    Session* _session = nullptr;
};

RegisterAccept

RegisterAccept는 "다음 클라이언트 한 명을 받을 준비"를 하는 함수다. 새 Session을 만들고, 그 session의 client socket을 AcceptEx에 넘긴다. listen socket은 이미 IOCP에 등록되어 있으므로, 이 accept 작업이 완료되면 완료 통지는 같은 IOCP 큐로 들어온다. 

void Listener::RegisterAccept(AcceptEvent* acceptEvent)
{
    Session* session = xnew<Session>();

    acceptEvent->Init();
    acceptEvent->SetSession(session);

    DWORD bytesReceived = 0;
    if (false == SocketUtils::AcceptEx(
        _socket,
        session->GetSocket(),
        session->_recvBuffer,
        0,
        sizeof(SOCKADDR_IN) + 16,
        sizeof(SOCKADDR_IN) + 16,
        OUT &bytesReceived,
        static_cast<LPOVERLAPPED>(acceptEvent)))
    {
        const int32 errCode = WSAGetLastError();
        if (errCode != WSA_IO_PENDING)
        {
            RegisterAccept(acceptEvent);
        }
    }
}

Worker thread와 Dispatch

GameServer는 여러 worker thread를 만들고 각 thread에서 계속 GIocpCore.Dispatch()를 호출한다. 이 함수는 내부적으로 GetQueuedCompletionStatus에서 대기한다. 큐에 완료 이벤트가 없으면 기다리고, 완료 이벤트가 들어오면 세 가지 정보를 받는다.

  • numOfBytes: 완료된 작업의 바이트 수
  • iocpObject: 등록 시 completion key로 넣어둔 객체 포인터
  • iocpEvent: Overlapped 포인터, 현재 구조에서는 IocpEvent*로 해석
bool IocpCore::Dispatch(uint32 timeoutMs)
{
    DWORD numOfBytes = 0;
    IocpObject* iocpObject = nullptr;
    IocpEvent* iocpEvent = nullptr;

    if (::GetQueuedCompletionStatus(
        _iocpHandle,
        OUT &numOfBytes,
        OUT reinterpret_cast<PULONG_PTR>(&iocpObject),
        OUT reinterpret_cast<LPOVERLAPPED*>(&iocpEvent),
        timeoutMs))
    {
        iocpObject->Dispatch(iocpEvent, numOfBytes);
    }

    return true;
}

왜 Listener::Dispatch로 가는가

listen socket을 IOCP에 등록할 때 completion key에 this, 즉 Listener*를 넣었다. 그래서 accept 완료 이벤트를 꺼내면 iocpObject는 Listener가 된다. IocpCore는 구체적인 타입을 몰라도 IocpObject 인터페이스만 보고 Dispatch를 호출할 수 있다. 

void Listener::Dispatch(IocpEvent* iocpEvent, int32 numOfBytes)
{
    ASSERT_CRASH(iocpEvent->GetType() == EventType::Accept);

    AcceptEvent* acceptEvent = static_cast<AcceptEvent*>(iocpEvent);
    ProcessAccept(acceptEvent);
}

ProcessAccept

ProcessAccept는 accept가 끝난 뒤 실제 연결을 완성하는 단계다. 먼저 SO_UPDATE_ACCEPT_CONTEXT로 accept된 client socket에 listen socket의 컨텍스트를 연결한다. 그 다음 getpeername으로 클라이언트 주소를 얻고, Session에 저장한다. 마지막으로 다시 RegisterAccept를 호출해서 다음 클라이언트 접속을 받을 준비를 한다. 

void Listener::ProcessAccept(AcceptEvent* acceptEvent)
{
    Session* session = acceptEvent->GetSession();
    if (false == SocketUtils::SetUpdateAcceptSocket(session->GetSocket(), _socket))
    {
        RegisterAccept(acceptEvent);
        return;
    }

    SOCKADDR_IN sockAddress;
    int32 sizeOfSockAddr = sizeof(sockAddress);
    if (SOCKET_ERROR == ::getpeername(session->GetSocket(), OUT reinterpret_cast<SOCKADDR*>(&sockAddress), &sizeOfSockAddr))
    {
        RegisterAccept(acceptEvent);
        return;
    }

    session->SetNetAddress(NetAddress(sockAddress));

    cout << "Client Connected!" << endl;

    RegisterAccept(acceptEvent);
}

흐름을 따라 읽는 순서

  1. CoreGlobal에서 Winsock과 확장 함수 포인터를 초기화한다.
  2. GameServer::main에서 Listener::StartAccept를 호출한다.
  3. Listener가 listen socket을 만들고 IOCP에 등록한다.
  4. bind, listen 후 AcceptEx를 걸어둔다.
  5. 클라이언트가 접속하면 accept 작업이 완료되고 IOCP 큐에 완료 이벤트가 들어간다.
  6. worker thread가 IocpCore::Dispatch에서 완료 이벤트를 꺼낸다.
  7. completion key로 받은 IocpObject*를 통해 Listener::Dispatch가 호출된다.
  8. AcceptEvent를 꺼내 ProcessAccept에서 연결 후처리를 한다.
  9. 다시 RegisterAccept를 호출해 다음 접속을 받을 준비를 한다.

오늘의 핵심 정리

  • IOCP는 socket에서 발생한 Overlapped 작업 완료를 한 큐로 모아준다.
  • CreateIoCompletionPort의 completion key 덕분에 완료 이벤트를 어떤 객체가 처리해야 하는지 알 수 있다.
  • AcceptEx는 먼저 걸어두는 작업이고, WSA_IO_PENDING은 정상적인 대기 상태다.
  • IocpEvent가 OVERLAPPED를 상속하므로 완료 시 이벤트 객체를 다시 복원할 수 있다.
  • Listener는 accept 완료를 처리한 뒤 곧바로 다음 accept를 다시 걸어 서버가 계속 접속을 받을 수 있게 한다.

논블로킹 소켓

관련 커밋: d45a83b 논블로킹 소켓

개념 한눈에 보기

논블로킹 모드에서는 소켓 함수가 조건이 만족될 때까지 기다리지 않고 즉시 돌아옵니다. 아직 처리할 수 없다면 WSAEWOULDBLOCK을 받게 되며, 이는 치명적 오류가 아니라 “나중에 다시 시도하라”는 신호입니다. 이 모델은 스레드 정지를 피하지만, 준비 상태를 효율적으로 감지하는 장치가 없으면 CPU를 낭비하기 쉽습니다.

블로킹과 논블로킹의 차이

Blocking recv데이터가 올 때까지 스레드가 멈춤
Non-blocking recv데이터가 없으면 즉시 WSAEWOULDBLOCK
장점한 스레드가 멈추지 않고 다른 일을 할 수 있음
주의무작정 루프를 돌면 busy polling이 됨

무엇을 배웠나

블로킹 소켓은 조건이 만족될 때까지 스레드를 멈추지만, 논블로킹 소켓은 즉시 반환합니다. 아직 처리할 수 없는 상태라면 WinSock은 WSAEWOULDBLOCK을 돌려줍니다.

논블로킹 전환

SOCKET listenSocket = ::socket(AF_INET, SOCK_STREAM, 0);
u_long on = 1;
::ioctlsocket(listenSocket, FIONBIO, &on);

accept 처리

while (true)
{
    SOCKET clientSocket = ::accept(listenSocket, (SOCKADDR*)&clientAddr, &addrLen);
    if (clientSocket == INVALID_SOCKET)
    {
        if (::WSAGetLastError() == WSAEWOULDBLOCK)
            continue;

        break;
    }

    cout << "Client Connected" << endl;
}

connect 처리

while (true)
{
    if (::connect(clientSocket, (SOCKADDR*)&serverAddr, sizeof(serverAddr)) == SOCKET_ERROR)
    {
        if (::WSAGetLastError() == WSAEWOULDBLOCK)
            continue;

        if (::WSAGetLastError() == WSAEISCONN)
            break;

        break;
    }
}

학습 포인트

논블로킹은 스레드를 멈추지 않는 대신 “지금 가능한가?”를 직접 계속 물어봐야 합니다. 이 커밋의 코드는 이해를 위해 continue로 반복 확인하지만, 실제 서버에서 이런 busy polling은 CPU를 낭비합니다. 그래서 다음 단계의 select, WSAEventSelect, Overlapped I/O 같은 준비/완료 통지 모델로 넘어갑니다.

send/recv에서의 핵심

  • recv가 SOCKET_ERROR이고 에러가 WSAEWOULDBLOCK이면 아직 읽을 데이터가 없는 상태입니다.
  • send도 송신 버퍼가 부족하면 WSAEWOULDBLOCK이 날 수 있습니다.
  • 논블로킹 send는 일부만 성공할 수 있으므로 전송한 바이트 수를 누적해야 합니다.
